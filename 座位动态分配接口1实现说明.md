# 座位可用性查询与动态分配 - 接口1实现说明

本文对应对话中“座位可用性查询与动态分配”第一个接口的实际实现情况，便于你在代码与文档之间对照。

---

## 一、代码实现概览

### 1.1 新增/修改的核心类

- **配置类**：`BusinessRulesProperties`
  - 新增 `SeatRules seat` 与 `TempLeaveRule`：  
    - `minDurationMinutes`、`maxDurationMinutes`、`slotStepMinutes`；  
    - `earliestBookingDays`、`latestBookingDays`；  
    - `noShowAfterMinutes`、`tempLeaveDefaultMinutes`；  
    - `tempLeaveLunch`、`tempLeaveDinner`（暂离规则，当前接口仅用到时长相关配置）。

- **实体类**：`SeatReservation`
  - 新增字段以对齐 `tb_seat_reservation` 表结构：  
    - `libraryId`：所属馆ID（冗余）；  
    - `borrowId`：关联借阅ID；  
    - `source`：预约来源（`WALK_IN` / `BORROW_PICKUP` 等）。

- **DTO**：
  - `SeatInfoDTO`：封装座位对外展示信息；
  - `SeatAvailabilityResultDTO`：封装查询结果（列表模式 + 自动分配模式）。

- **服务层**：`SeatAvailabilityService`
  - 核心方法：  
    `public SeatAvailabilityResultDTO queryAvailableSeats(Long libraryId, LocalDate reservationDate, LocalTime startTime, LocalTime endTime, Long areaId, Boolean hasPower, Boolean autoAssign)`
  - 职责：
    - 参数与时间范围校验（基于 `SeatRules` 配置）；
    - 按馆过滤区域 → 按区域与状态过滤座位 → 按预约记录过滤时间冲突；
    - 按“有电源优先 → 楼层升序 → 区域名升序 → 座位号升序”排序；
    - 根据 `autoAssign` 决定返回列表或推荐座位。

- **控制器**：`SeatController`
  - 新增接口：`GET /api/seat/available`
  - 使用 Swagger 注解标注参数与描述，返回 `Result<SeatAvailabilityResultDTO>`。

## 二、时间与日期校验逻辑

在 `SeatAvailabilityService.validateTimeRange` 中：

1. **预约日期范围**：
   - 使用 `earliestBookingDays` 与 `latestBookingDays`，对比 `LocalDate.now()` 与 `reservationDate` 的天数差；  
   - 若超出范围，抛出 `IllegalArgumentException("预约日期不在允许范围内")`。

2. **起止时间合法性**：
   - 要求 `endTime > startTime`，否则抛出 `IllegalArgumentException("结束时间必须大于开始时间")`。

3. **时长与步长**：
   - 使用 `ChronoUnit.MINUTES.between(startTime, endTime)` 计算时长；  
   - 要求：
     - `minDurationMinutes <= minutes <= maxDurationMinutes`；  
     - `minutes % slotStepMinutes == 0`，否则抛出 `"预约时长必须是X分钟的整数倍"`。

异常统一在 `SeatController` 中捕获，转换为 `Result.fail(400, message)`。

## 三、候选座位与时间冲突过滤

### 3.1 候选区域与座位

1. 根据 `libraryId` 调用 `SeatAreaMapper.selectByLibraryId(libraryId)`；
2. 过滤出 `status='OPEN'` 的区域，避免关闭或维护中的区域参与选座；
3. 若传入 `areaId`：
   - 且 `areaId` 属于当前馆的区域列表：仅在该区域下选座；
   - 否则直接返回空结果（`total=0`），防止跨馆或越界区域选座；
4. 使用 `SeatMapper` + `LambdaQueryWrapper`：
   - `area_id IN (:areaIds)`；
   - `status='AVAILABLE'`；
   - 若 `hasPower=true`，则加上 `has_power=1` 条件。

### 3.2 时间冲突过滤

1. 收集候选座位的 `seatId` 列表；
2. 在 `tb_seat_reservation` 中查出：
   - `seat_id IN (:seatIds)`；
   - `reservation_date = :reservationDate`；
   - `status='ACTIVE'`；
3. 将查询结果按 `seatId` 分组，构建 `Map<Long, List<SeatReservation>>`；
4. 对每个候选座位：
   - 遍历其 `ACTIVE` 预约记录，判断是否与 `[startTime, endTime]` 重叠；  
   - **不重叠条件**：`newEnd <= existStart` 或 `newStart >= existEnd`；  
   - 若存在任何一个记录与新时段重叠，则排除此座位。

## 四、排序与自动分配逻辑

过滤后得到的候选座位列表，在服务层做排序：

1. 有电源优先：`hasPower` 从大到小（1 在前，0 在后）；
2. 楼层升序：`floor` 小的在前（例如 1 楼优先于 3 楼）；
3. 区域名称升序：`areaName` 字典序；
4. 座位编号升序：`seatNo` 字典序。

随后根据 `autoAssign`：

- 若 `autoAssign == null` 或 `true`：
  - 设置 `availableTotal = candidates.size()`；
  - 若不为空，`assignedSeat = candidates.get(0)`；
- 若 `autoAssign == false`：
  - 设置 `total = candidates.size()`；
  - `seats = candidates`。

在 `SeatController` 中：

- 对自动分配模式下 `availableTotal == 0`，以及列表模式下 `total == 0` 的情况，统一返回 `code=200`，`message="当前条件下无可用座位"`，由前端根据数量字段判断有无座位。
