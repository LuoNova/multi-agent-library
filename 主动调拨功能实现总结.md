## 三、核心算法

### 3.1 调拨触发条件

**需求馆判定**：
- 库存不足（< 2本）
- 需求旺盛（近30天借出次数 > 20次 或 预约排队人数 > 3）

**供应馆判定**：
- 库存充足（> 5本）
- 利用率低（近30天借出次数 < 10次）

### 3.2 调拨数量计算

```
调拨数量 = min(
    源馆可用库存 × 0.3,  // 最多调出源馆30%的库存
    目标馆需求缺口,       // 目标馆需要的数量
    5                     // 单次最多调拨5本
)
```

### 3.3 优先级评分算法

```
调拨优先级评分 = 
    目标馆需求度 × 0.4 +    // 近30天借出次数、预约排队人数
    源馆富余度 × 0.3 +      // 库存数量、利用率
    距离成本 × 0.2 +        // 馆与馆之间的距离
    历史调拨频率 × 0.1      // 历史调拨成功率
```

### 3.4 审批流程（混合模式）

- **小批量调拨（≤ 3本）**：自动审批并执行
- **大批量调拨（> 3本）**：等待人工审批

### 11.2 问题2：调拨建议可能重复生成

**问题描述**：
- 每次执行库存平衡分析时，可能对同一本书生成多条调拨建议
- 如果建议未被审批，下次分析又会生成新的建议

**修复方案**：

在`InventoryBalanceServiceImpl.analyzeBiblio`方法中添加去重检查：

```java
//检查是否已存在相同的待审批建议（去重）
LambdaQueryWrapper<TransferSuggestion> wrapper = new LambdaQueryWrapper<>();
wrapper.eq(TransferSuggestion::getBiblioId, biblioId)
       .eq(TransferSuggestion::getToLibraryId, demand.getLibraryId())
       .eq(TransferSuggestion::getFromLibraryId, supply.getLibraryId())
       .eq(TransferSuggestion::getStatus, "PENDING");

if (suggestionMapper.selectCount(wrapper) > 0) {
    log.info("已存在待审批的调拨建议：书目{}，从馆{}到馆{}，跳过",
            biblioId, supply.getLibraryId(), demand.getLibraryId());
    continue;
}
```

## 十三、审批流程问题修复

### 13.1 问题描述

**现象**：
- 审批通过后，建议状态变为`APPROVED`
- 但调用`executeSuggestion`时，报错"调拨建议状态不正确: APPROVED"
- 导致调拨单未创建

**根本原因**：
- `approveSuggestion`方法先更新状态为`APPROVED`
- 然后调用`executeSuggestion`方法
- 但`executeSuggestion`方法只允许`PENDING`状态执行

### 13.2 修复方案

**修改`executeSuggestion`方法**，允许`PENDING`或`APPROVED`状态执行：

```java
//允许PENDING或APPROVED状态执行
if (!"PENDING".equals(suggestion.getStatus()) && !"APPROVED".equals(suggestion.getStatus())) {
    log.error("调拨建议状态不正确: {}", suggestion.getStatus());
    return;
}
```

### 13.3 修复后的状态流转

**自动审批模式**：
- PENDING → EXECUTED（直接执行）

**人工审批模式**：
- PENDING → APPROVED → EXECUTED（审批后执行）
