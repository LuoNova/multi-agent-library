# 座位动态分配 - 功能2「创建座位预约」设计说明

本文是对对话中“第二个接口”（创建座位预约）的设计整理版本，已同步追加到项目根目录下的 `功能文档.md` 中，这里单独成文便于查阅。

---

## 一、功能概述

- **核心目标**：在用户已经选定具体座位（或由系统自动分配出推荐座位）并确认预约时间段后，为用户创建一条 `tb_seat_reservation` 记录，并将对应座位标记为已占用。  
- **位置**：`Part2 座位动态分配` 下的 **功能2：创建座位预约**。  
- **与功能1关系**：  
  - 功能1负责“算出哪些座位在给定时间段内是可用的”；  
  - 功能2负责“基于选定的座位 + 时间段真正写预约记录并占用座位”。

---

## 二、业务规则与流程

### 2.1 参数与时间合法性校验

- 基于 `BusinessRulesProperties.SeatRules` 中的配置：
  - `earliestBookingDays` / `latestBookingDays`：控制 `reservationDate` 相对今天的可选范围；  
  - `minDurationMinutes` / `maxDurationMinutes`：控制单次预约的最小/最大时长；  
  - `slotStepMinutes`：控制时长必须是该步长的整数倍。
- 具体校验规则：
  - `endTime > startTime`，否则直接拒绝；
  - `durationMinutes = endTime - startTime` 必须满足：
    - `minDurationMinutes <= durationMinutes <= maxDurationMinutes`；  
    - `durationMinutes % slotStepMinutes == 0`。
- 开放时间校验（待与你最终确认）：
  - 优先使用 `tb_seat_area.open_time` 作为区域开放时间；  
  - 若区域 `open_time` 为空，则回退使用 `tb_library.open_time`；  
  - 预约时段需整体落在开放时间内。

### 2.2 座位与区域状态校验

- 校验 `seatId` 对应的 `tb_seat` 是否存在，且 `status='AVAILABLE'`；  
- 通过 `tb_seat.area_id` 反查 `tb_seat_area`，确认：
  - 该区域属于请求参数中的 `libraryId`，避免跨馆选座；  
  - 区域 `status='OPEN'`。
- 不满足则返回失败（例如“座位不存在或不可用”“区域未开放或不属于该馆”）。

### 2.3 时间冲突与用户冲突校验

- **座位维度冲突**（必做）：  
  - 在 `tb_seat_reservation` 中按 `(seat_id, reservation_date, status='ACTIVE')` 查询已有预约；  
  - 判断 `[startTime, endTime]` 是否与任一已存在的 `[existStart, existEnd]` 重叠：  
    - 不重叠条件：`newEnd <= existStart` 或 `newStart >= existEnd`；  
    - 若存在任一条不满足该条件，则视为冲突，拒绝创建。

- **用户维度冲突**（建议默认开启，待你确认）：  
  - 同一用户在同一馆、同一天内的 **ACTIVE 预约** 不允许时间段重叠；  
  - 即按 `(user_id, library_id, reservation_date, status='ACTIVE')` 查出该用户当日已有预约，若与新时段重叠，则拒绝：  
    - 目的：防止一个用户同一时间段占用多个座位；  
    - 如你希望支持“帮同学占座”，则可以关掉这一条，只做“座位维度”的冲突校验。

### 2.4 持久化创建预约记录

- 在 `tb_seat_reservation` 插入新记录：
  - 必备字段：  
    - `seat_id`、`user_id`、`reservation_date`、`start_time`、`end_time`；  
    - `status='ACTIVE'`；  
    - `check_in_time=NULL`、`create_time=NOW()`；  
    - `library_id`：由 `seat.area_id -> seat_area.library_id` 反查写入。
  - 可选字段：  
    - `borrow_id`：当用户从“借书/取书推荐”入口进入并携带 `borrowId` 时写入；  
    - `source`：`WALK_IN` / `BORROW_PICKUP` 等，具体赋值规则见下。

### 2.5 更新座位状态为 OCCUPIED

- 插入预约成功后，将对应 `tb_seat.status` 更新为 `OCCUPIED`：  
  - 表示该座位在当前时间段内**已经被预约/占用**；  
  - 后续在“取消预约 / NO_SHOW / 结束使用 / 时段结束”时恢复为 `AVAILABLE`。
- 建议利用事务保证一致性：
  - 同一事务内进行“冲突校验 → 插入预约 → 更新座位状态”；  
  - 任一步失败则整体回滚，避免出现“座位变成 OCCUPIED 但无预约记录”的异常状态。

### 2.6 返回结果

- 成功时：
  - `code=200`；  
  - `message="座位预约创建成功"`；  
  - `data` 中包含 `reservationId`、座位基本信息（区域、楼层、座位号、电源）、预约日期与时段、`source` 等。
- 失败时：
  - `code=400`；  
  - `message` 描述具体原因，如“该时间段座位已被占用”“您在该时间段已预约其他座位”“参数非法”等；  
  - `data=null`。

## 三、接口视图（功能层面）

### 3.1 路径与方法

- `POST /api/seat/reservation`

### 3.2 请求体字段（JSON）

| 字段名           | 类型   | 必填 | 说明                                                       |
|------------------|--------|------|------------------------------------------------------------|
| `userId`         | Long   | 是   | 用户ID                                                     |
| `seatId`         | Long   | 是   | 目标座位ID                                                 |
| `libraryId`      | Long   | 是   | 馆ID，用于校验座位所属馆是否匹配                          |
| `reservationDate`| String | 是   | 预约日期 `yyyy-MM-dd`，需在允许范围内                    |
| `startTime`      | String | 是   | 开始时间 `HH:mm`                                          |
| `endTime`        | String | 是   | 结束时间 `HH:mm`，需大于 `startTime`                      |
| `borrowId`       | Long   | 否   | 关联借阅ID（从借书/取书推荐入口进入时填写）              |
| `source`         | String | 否   | 预约来源：`WALK_IN` / `BORROW_PICKUP` 等，缺省时可后端推断 |

### 3.3 响应示例（成功）

```json
{
  "code": 200,
  "message": "座位预约创建成功",
  "data": {
    "reservationId": 5001,
    "userId": 1,
    "libraryId": 1,
    "seatId": 1001,
    "areaId": 1,
    "areaName": "一楼自习区A",
    "floor": 1,
    "seatNo": "A-01",
    "hasPower": true,
    "reservationDate": "2026-03-10",
    "startTime": "09:00",
    "endTime": "11:00",
    "source": "WALK_IN"
  }
}
```

### 3.4 响应示例（失败）

```json
{
  "code": 400,
  "message": "该时间段座位已被占用,请更换时间或座位",
  "data": null
}
```

或（用户冲突）：

```json
{
  "code": 400,
  "message": "您在该时间段已预约其他座位,请先取消或调整时间",
  "data": null
}
```

## 四、数据库影响

1. **写入 `tb_seat_reservation`**：  
   - 新增一条 `ACTIVE` 状态的预约记录，填充 `library_id`（冗余）、`borrow_id`（可选）、`source` 等字段；
2. **更新 `tb_seat`**：  
   - 将对应 `seat_id` 的 `status` 更新为 `OCCUPIED`；
3. **不修改结构**：  
   - 不新增任何新表或字段，全部在现有结构基础上实现。

---

## 五、待确认要点

1. **同一用户同一时间段是否允许多座预约**  
   - 当前建议：**不允许**，防止一人多占座位；  
   - 如你有“帮人占座”的刚性需求，可以将用户维度校验关闭，仅按座位维度控制。

2. **开放时间校验的精细规则**  
   - 是否采用“同时满足馆与区域开放时间”的 AND 规则，还是以区域为准、区域为空时回退到馆？  
   - `open_time` 为空时是否视为“全天开放”？

3. **`source` 由前端传还是后端推断**  
   - 方案 A：前端显式传入 `source`，后端只负责保存；  
   - 方案 B：后端根据 `borrowId` 是否为空自动推断（有借阅ID → `BORROW_PICKUP`，否则 `WALK_IN`）。

4. **错误提示粒度**  
   - 当既有用户自身冲突又有座位冲突时，是分别提示还是统一成一个笼统错误文案？  
   - 建议优先提示用户自身冲突，更利于用户决策。

你确认完上述要点后，我就可以按照本文档中的设计去实现 `SeatReservation` 的创建接口与对应的 Service 逻辑，并在《功能测试指导.md》中追加该接口的测试步骤。
