# 多智能体图书馆管理系统 - 功能文档

## 功能1：**智能借书与资源调度（Smart Borrowing）**

### 功能描述

用户发起借书请求后，系统通过多智能体协作自动选择最优策略：

- **本地借阅（LOCAL_LOAN）**：目标馆有库存时立即借出
- **跨馆调拨（TRANSFER_PROVIDE）**：目标馆无库存但其他馆有，自动协商调拨并预留24小时
- **预约排队（RESERVATION）**：全馆无库存时创建预约，书归还后自动优先分配

系统内置权限检查（最大借阅数、重复借书限制），确保业务合规。

### 参与智能体

| 智能体              | 角色                 | 职责                                             |
| :------------------ | :------------------- | :----------------------------------------------- |
| UserDemandAgent     | Manager（协调者）    | 接收请求、权限检查、策略选择、调拨执行、预约管理 |
| InventoryAgent-LibN | Contractor（参与者） | 评估本地库存、计算投标评分（距离+压力）、响应CFP |

### 业务流程（时序）

1. **权限检查**：UserDemandAgent检查用户状态（是否冻结、是否达到最大借阅数、是否重复借书）
2. **库存评估**：向所有InventoryAgent发送CFP（Call for Proposal），收集各馆库存和投标评分
3. **策略选择**：
   - **本地借阅**：目标馆有库存 → 直接分配副本，创建借阅记录
   - **跨馆调拨**：目标馆无库存但其他馆有 → 选择最优来源馆，创建调拨记录和运输中借阅记录
   - **预约兜底**：全馆无库存 → 创建预约记录（策略C）
4. **执行与预留**：
   - 策略A：创建借阅记录（`BORROWING`），副本状态改为`BORROWED`
   - 策略B：执行调拨（`tb_book_transfer`），创建借阅记录（`TRANSFERING`），等待书籍调拨完成
   - 策略C：创建预约记录（`PENDING`，3天有效）
5. **结果返回**：返回对应策略的执行结果

### 核心算法

**投标评分公式**（仅策略B使用）：

```plain
score = 80 - (distance(km) × 10) - (pending_reservation_count × 5)
```

## 功能2：**还书自动分配与调拨履约（Smart Return & Auto-Fulfillment）**

### 功能描述

用户还书时，系统自动检查该书目是否有人预约，实现**"还书即分配"**的零等待体验：

- **就地预留（LOCAL_RESERVE）**：预约者期望取书馆与还书馆相同，直接锁定副本24小时
- **跨馆调拨（TRANSFER_FULFILL）**：预约者在其他分馆，触发物流调拨，到馆后锁定24小时
- **预约超期释放（RESERVE_EXPIRE）**：24小时内未取书，自动释放给下一位预约者或转为可借
- **信用分计算（CREDIT_PENALTY）**：逾期归还扣除信用分（每天2分，最高20分），低于60分冻结账号

### 参与智能体

| 智能体                  | 角色                   | 职责                                                 |
| :---------------------- | :--------------------- | :--------------------------------------------------- |
| ReturnAgent             | Manager（协调者）      | 接收还书请求、信用分计算、触发预约队列检查、协调调拨 |
| TransferAgent           | Contractor（物流履约） | 管理调拨状态流转、运输完成回调、库存同步             |
| ReservationMonitorAgent | Monitor（监控者）      | 定时扫描预留超期记录、自动释放、级联分配             |

### 业务流程（时序）

1. **权限校验**：ReturnAgent确认借书记录存在、借书人身份匹配、状态为BORROWING
2. **基础还书**：更新副本为AVAILABLE、关闭借阅记录、减少用户借阅计数、增加还书馆库存
3. **信用评估**：检查是否逾期，扣除信用分（如逾期），冻结账号（如信用分<60）
4. **预约队列遍历**：按FIFO顺序检查`tb_book_reservation`中该`biblio_id`的PENDING记录：
   - 如果预约者已达借阅上限/账号冻结，标记CANCELED，继续下一个
   - 直到找到有效预约者或队列为空
5. **策略执行**：
   - **就地预留**：副本状态改为RESERVED，创建借阅记录（状态RESERVED，24小时过期）
   - **跨馆调拨**：创建调拨记录（状态IN_TRANSIT），副本状态改为IN_TRANSFER，通知物流系统
6. **调拨到货回调**：TransferAgent接收物流系统回调，更新调拨状态为COMPLETED，副本移至目标馆，创建借阅记录（状态RESERVED，24小时过期），通知预约者取书
7. **预留超期处理**：ReservationMonitorAgent每小时扫描，将过期RESERVED记录改为EXPIRED，副本释放，触发下一次预约队列检查

### 核心算法

**预约者优先级评分**（仅用于 Debug 日志，实际为 FIFO）：

```plain
priority = reserve_time（早者优先）
```

**逾期信用分计算**：

```plain
penalty = min(overdue_days × 2, 20)
new_credit = max(0, old_credit - penalty)
if new_credit < 60 then status = FROZEN
```

## 功能3：调拨完成回调（Transfer Complete Callback）

### 功能描述

当图书通过物流从源馆运输到目标馆后，由物流人员或馆员确认收货，系统通过此接口完成调拨闭环。根据调拨类型（借书调拨、还书调拨）采取不同策略：

- **借书调拨**：更新已有的运输中借阅记录为预留状态，启动24小时取书倒计时
- **还书调拨**：为预约用户创建新的预留借阅记录，启动24小时取书倒计时

### 参与智能体

| 智能体        | 角色               | 职责                                                  |
| :------------ | :----------------- | :---------------------------------------------------- |
| TransferAgent | Executor（执行者） | 验证调拨单、更新库存、创建/更新借阅记录、同步用户计数 |

### 业务流程（时序）

1. **接收回调**：TransferController接收调拨完成请求，校验参数合法性
2. **状态校验**：检查调拨单状态是否为`IN_TRANSIT`（运输中）
3. **更新调拨单**：将调拨单状态改为`COMPLETED`，记录完成时间
4. **更新副本位置**：将副本物理位置更新到目标馆，状态改为`RESERVED`（预留）
5. **更新库存统计**：目标馆该书的库存+1
6. **处理借阅记录**：
   - **借书调拨**：找到运输中的借阅记录，更新为`RESERVED`，设置24小时过期时间
   - **还书调拨**：为预约用户创建新的借阅记录（`RESERVED`），设置24小时过期时间，用户借阅计数+1
7. **发送通知**：向用户发送"图书已到达，请及时取书"通知

## 功能4：**取书确认（Pickup Confirmation）**

### 功能描述

用户到馆后确认取书,完成借阅流程。系统验证用户身份、预留状态和预留时间,将借阅状态从`RESERVED`(已预留)转换为`BORROWING`(借阅中),正式开始借阅计时。

### 业务规则

- **仅限本人取书**: 必须是预约人本人,不支持代取书
- **24小时有效期**: 必须在预留期内取书,超期需重新预约
- **状态校验**: 只有`RESERVED`状态的记录才能取书
- **借阅期限**: 取书确认后开始计算借阅期限(默认30天)

### 执行流程

1. **接收请求**: PickupController接收取书确认请求
2. **身份验证**: 验证用户是否为预约人本人
3. **状态校验**: 检查借阅状态是否为`RESERVED`
4. **时间校验**: 检查是否在24小时预留期内
5. **状态更新**: 
   - 借阅记录: `RESERVED` → `BORROWING`
   - 图书副本: `RESERVED` → `BORROWED`
6. **时间记录**: 记录实际取书时间,计算应还日期
7. **结果返回**: 返回取书成功信息和借阅详情

## 功能5：**主动调拨与库存平衡（Proactive Transfer & Inventory Balance）**

### 功能描述

系统基于馆藏书目统计表（`tb_library_biblio_stats`）的借阅热度分析，自动触发低利用率馆向高需求馆的主动调拨，实现库存动态平衡，提高资源利用率。

系统支持两种审批模式：
- **自动审批**：调拨数量≤3本时，系统自动审批并执行
- **人工审批**：调拨数量>3本时，生成调拨建议等待馆员审批

### 参与智能体

| 智能体 | 角色 | 职责 |
| :----- | :--- | :--- |
| InventoryBalanceAgent | 库存平衡代理 | 定期分析库存、生成调拨建议、执行调拨 |

### 执行流程

1. **库存分析**：
   - 查询所有图书馆的书目统计信息
   - 识别需求旺盛的馆（目标馆）：库存<2本 且 （近30天借出>20次 或 预约排队>3人）
   - 识别库存充足的馆（源馆）：库存>5本 且 近30天借出<10次

2. **生成调拨建议**：
   - 为每个需求馆寻找最佳供应馆
   - 计算调拨数量：min(源馆库存×30%, 目标馆需求缺口, 5本)
   - 计算优先级评分：目标馆需求度×0.4 + 源馆富余度×0.3 + 距离成本×0.2 + 历史调拨频率×0.1
   - 生成调拨建议并保存到`tb_transfer_suggestion`表

3. **审批执行**：
   - 小批量调拨（≤3本）：自动审批并执行
   - 大批量调拨（>3本）：通知馆员审批（TODO: 管理员系统未实现，暂时控制台打印）
   - 审批通过后创建调拨单

4. **调拨执行**：
   - 查找源馆的可用副本
   - 创建调拨单（状态：IN_TRANSIT）
   - 更新副本状态（IN_TRANSIT）
   - 更新调拨建议状态（EXECUTED）

### 核心算法

**调拨数量计算公式**：

```plain
调拨数量 = min(
    源馆可用库存 × 0.3,  // 最多调出源馆30%的库存
    目标馆需求缺口,       // 目标馆需要的数量
    5                     // 单次最多调拨5本
)
```

**优先级评分公式**：

```plain
优先级评分 = 
    目标馆需求度 × 0.4 +    // 近30天借出次数、预约排队人数
    源馆富余度 × 0.3 +      // 库存数量、利用率
    距离成本 × 0.2 +        // 馆与馆之间的距离
    历史调拨频率 × 0.1      // 历史调拨成功率
```

## 功能6：**调拨进度跟踪与通知服务（Transfer Progress & Notification）**

### 功能描述

为调拨功能提供进度查询和通知服务,提升用户体验：

- **调拨进度跟踪**：查询单个调拨、批量调拨的实时进度,包括进度百分比、时间信息等
- **调拨通知服务**：发送调拨发起、到达通知(仅用户请求调拨)
- **延迟检测告警**：定时检测超时调拨并控制台打印告警

系统采用简化设计,不使用Redis缓存和事件驱动,直接同步调用,简单高效。

### 参与组件

| 组件 | 角色 | 职责 |
| :--- | :--- | :--- |
| TransferProgressController | Controller | 提供进度查询接口 |
| TransferProgressService | Service | 进度计算和数据组装 |
| TransferNotificationService | Service | 发送调拨通知 |
| TransferDelayCheckTask | Task | 定时检测延迟调拨 |

### 核心流程

#### 1. 调拨进度查询流程

1. **接收请求**：用户或管理员调用进度查询接口
2. **查询数据**：从数据库查询调拨记录、调拨单、图书、馆信息
3. **计算进度**：根据状态和时间计算进度百分比
   - PENDING: 0%
   - IN_TRANSIT: (已用时间/预计时间) × 100,最大95%
   - COMPLETED: 100%
   - CANCELED: 0%
4. **组装数据**：构建响应DTO,包含图书信息、馆信息、时间信息等
5. **返回结果**：返回进度详情

#### 2. 调拨通知流程

1. **触发通知**：调拨创建或完成时调用通知服务
2. **判断类型**：检查调拨原因(USER_REQUEST/INVENTORY_BALANCE)
3. **发送通知**：
   - USER_REQUEST: 发送站内信给目标用户
   - INVENTORY_BALANCE: 不发送通知(系统内部操作)
4. **记录通知**：写入`tb_notification`表

#### 3. 延迟检测流程

1. **定时触发**：每小时执行一次(Cron: `0 0 * * * ?`)
2. **查询延迟**：查询状态为IN_TRANSIT且超过预计到达时间的调拨记录
3. **打印告警**：控制台输出详细告警信息(调拨ID、图书、源馆、目标馆、延迟时长)

### 核心算法

**进度百分比计算公式**：

```plain
progress = (usedTime / estimatedTime) × 100
最大值: 95% (到达后才会变为100%)
```

**时长格式化规则**：
- 小于1小时: "X分钟"
- 1-24小时: "X小时Y分钟"
- 大于24小时: "X天Y小时"

## 功能7（补充）：Phase A一致性闭环增强（2026-03-09）

>本补充用于说明 Phase A 的关键一致性修复与增强，重点解决“调拨时间字段不一致、预留超期不闭环、用户无法查询个人调拨列表”的问题。

### 7.1 调拨时间字段统一（进度/延迟检测可靠）

#### 字段语义约定
- `tb_book_transfer.request_time`:调拨发起时间
- `tb_book_transfer.estimated_arrival_time`:预计到达时间(用于进度计算、延迟检测)
- `tb_book_transfer.actual_arrival_time`:实际到达时间(用于进度展示)
- `tb_book_transfer.complete_time`:调拨完成处理时间(与实际到达一致写入)

#### 写入时机
- 创建调拨(`TransferService.createTransfer`)时写入:
  - `request_time`
  - `estimated_arrival_time`
- 调拨完成(`TransferService.completeTransfer`)时写入:
  - `actual_arrival_time`
  - `complete_time`

#### 影响范围
- 调拨进度查询接口:`GET /api/transfer/progress/{transferId}`
- 延迟检测定时任务:`TransferDelayCheckTask`(按`estimated_arrival_time`判断超时)

---

### 7.2 预留超期释放闭环(释放+继续兑现下一预约者)

#### 功能描述
当借阅记录处于`RESERVED`且超过`pickup_deadline`仍未取书时，系统自动执行:
1. 借阅记录:`RESERVED -> CANCELLED`
2. 副本状态:`RESERVED -> AVAILABLE`
3. 用户借阅计数:`current_borrow_count -1`
4. 用户信用分:`credit_score -5`(低于60分冻结账号)
5. 自动尝试兑现同书目的下一位`PENDING`预约者(就地预留或跨馆调拨)

#### 关键实现点
- 新增`ReservationFulfillmentService`统一负责“为AVAILABLE副本兑现下一位预约者”
- `ReservationExpireTask`在释放副本后调用兑现服务，实现闭环
- 预留阶段的`tb_book_borrow`必须写入:
  - `reserved_time`
  - `pickup_deadline`
  - `pickup_library_id`
  - `due_time`(承载预留到期时间,取书确认后会覆盖为30天应还期)

提供用户视角的调拨列表查询，便于用户查看:
