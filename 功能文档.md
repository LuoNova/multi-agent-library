## 功能1：**智能借书与资源调度（Smart Borrowing）**

### 功能描述

用户发起借书请求后，系统通过多智能体协作自动选择最优策略：

- **本地借阅（LOCAL_LOAN）**：目标馆有库存时立即借出
- **跨馆调拨（TRANSFER_PROVIDE）**：目标馆无库存但其他馆有，自动协商调拨并预留24小时
- **预约排队（RESERVATION）**：全馆无库存时创建预约，书归还后自动优先分配

系统内置权限检查（最大借阅数、重复借书限制），确保业务合规。

### 参与智能体

| 智能体              | 角色                 | 职责                                             |
| :------------------ | :------------------- | :----------------------------------------------- |
| UserDemandAgent     | Manager（协调者）    | 接收请求、权限检查、策略选择、调拨执行、预约管理 |
| InventoryAgent-LibN | Contractor（参与者） | 评估本地库存、计算投标评分（距离+压力）、响应CFP |

### 业务流程（时序）

1. **权限检查**：UserDemandAgent检查用户状态、最大借阅数、是否已借该书
2. **策略选择**：
   - **本地优先**：检查`preferredLibraryId`是否有`AVAILABLE`副本 → 直接借书（策略A）
   - **跨馆协商**：本地无库存 → 广播CFP → 收集投标 → 选择最优馆调拨（策略B）
   - **预约兜底**：全馆无库存 → 创建`tb_book_reservation`记录（策略C）
3. **执行与预留**：
   - 策略A：创建借阅记录（`BORROWING`），副本状态改为`BORROWED`
   - 策略B：执行调拨（`tb_book_transfer`），创建借阅记录（`TRANSFERING`），等待书籍调拨完成
   - 策略C：创建预约记录（`PENDING`，3天有效）
4. **结果返回**：返回对应策略的执行结果

### 核心算法

**投标评分公式**（仅策略B使用）：

```plain
score = 80 - (distance(km) × 10) - (pending_reservation_count × 5)
```

## 功能2：**还书自动分配与调拨履约（Smart Return & Auto-Fulfillment）**

### 功能描述

用户还书时，系统自动检查该书目是否有人预约，实现**"还书即分配"**的零等待体验：

- **就地预留（LOCAL_RESERVE）**：预约者期望取书馆与还书馆相同，直接锁定副本24小时
- **跨馆调拨（TRANSFER_FULFILL）**：预约者在其他分馆，触发物流调拨，到馆后锁定24小时
- **预约超期释放（RESERVE_EXPIRE）**：24小时内未取书，自动释放给下一位预约者或转为可借
- **信用分计算（CREDIT_PENALTY）**：逾期归还扣除信用分（每天2分，最高20分），低于60分冻结账号

### 参与智能体

| 智能体                  | 角色                   | 职责                                                 |
| :---------------------- | :--------------------- | :--------------------------------------------------- |
| ReturnAgent             | Manager（协调者）      | 接收还书请求、信用分计算、触发预约队列检查、协调调拨 |
| TransferAgent           | Contractor（物流履约） | 管理调拨状态流转、运输完成回调、库存同步             |
| ReservationMonitorAgent | Monitor（监控者）      | 定时扫描预留超期记录、自动释放、级联分配             |

### 业务流程（时序）

1. **权限校验**：ReturnAgent确认借书记录存在、借书人身份匹配、状态为BORROWING
2. **基础还书**：更新副本为AVAILABLE、关闭借阅记录、减少用户借阅计数、增加还书馆库存
3. **信用评估**：检查是否逾期，扣除信用分（如逾期），冻结账号（如信用分<60）
4. **预约队列遍历**：按FIFO顺序检查`tb_book_reservation`中该`biblio_id`的PENDING记录：
   - 如果预约者已达借阅上限/账号冻结，标记CANCELED，继续下一个
   - 直到找到有效预约者或队列为空
5. **策略执行**：
   - **就地预留**：副本状态改为RESERVED，创建借阅记录（状态RESERVED，24小时过期）
   - **跨馆调拨**：创建调拨记录（状态IN_TRANSIT），副本状态改为IN_TRANSFER，通知物流系统
6. **调拨到货回调**：TransferAgent接收物流系统回调，更新调拨状态为COMPLETED，副本移至目标馆，创建借阅记录（状态RESERVED，24小时过期），通知预约者取书
7. **预留超期处理**：ReservationMonitorAgent每小时扫描，将过期RESERVED记录改为EXPIRED，副本释放，触发下一次预约队列检查

### 核心算法

**预约者优先级评分**（仅用于 Debug 日志，实际为 FIFO）：

```plain
priority = reserve_time（早者优先）
```

**逾期信用分计算**：

```plain
penalty = min(overdue_days × 2, 20)
new_credit = max(0, old_credit - penalty)
if new_credit < 60 then status = FROZEN
```

## 功能3：调拨完成回调（Transfer Complete Callback）

### 功能描述

当图书通过物流从源馆运输到目标馆后，由物流人员或馆员确认收货，系统通过此接口完成调拨闭环。根据调拨类型（借书调拨 、还书调拨）采取不同策略：

- **借书调拨**：更新已有的运输中借阅记录为预留状态，启动24小时取书倒计时
- **还书调拨**：为预约用户创建新的预留借阅记录，启动24小时取书倒计时

### 参与智能体

| 智能体        | 角色               | 职责                                                  |
| :------------ | :----------------- | :---------------------------------------------------- |
| TransferAgent | Executor（执行者） | 验证调拨单、更新库存、创建/更新借阅记录、同步用户计数 |

### 业务流程（时序）

1. **接收回调**：TransferController接收调拨完成请求，校验参数合法性
2. **状态校验**：TransferService查询调拨单，确认状态为`IN_TRANSIT`（运输中）
3. **更新调拨单**：将调拨状态改为`COMPLETED`，记录实际到达时间
4. **实物上架**：更新图书副本物理位置到目标馆，状态改为`RESERVED`（已预留），设置默认位置为"馆X-调拨暂存区"
5. **库存同步**：目标馆库存统计+1（物理到货）
6. **策略分支**：
   - **分支A（借书调拨）**：查询`tb_book_borrow`中`TRANSFERRING`状态的记录，更新为`RESERVED`，设置24小时过期时间（用户计数已在发起时增加，此处不再增加）
   - **分支B（还书调拨）**：查询`tb_book_reservation`中`FULFILLED`状态的记录，创建新的`RESERVED`借阅记录，用户借阅计数+1
7. **结果返回**：返回创建的借阅记录ID和预留过期时间

### 核心算法

**调拨类型判定逻辑**：

```java
if (存在TRANSFERRING状态的Borrow记录) {
    //借书调拨：发起时创建了Borrow记录，只需更新状态
    更新Borrow为RESERVED;
} else {
    //还书调拨：发起时创建了Reservation，需新建Borrow并增加用户计数
    创建Borrow(RESERVED);
    用户计数+1;
}
```

## 功能5：**主动调拨与库存平衡（Proactive Transfer & Inventory Balance）**

### 功能描述

系统基于馆藏书目统计表（`tb_library_biblio_stats`）的借阅热度分析，自动触发低利用率馆向高需求馆的主动调拨，实现库存动态平衡，提高资源利用率。

### 业务规则

- **触发条件**：
  - 目标馆库存不足（< 2本）
  - 目标馆需求旺盛（近30天借出次数 > 20次 或 预约排队人数 > 3）
  - 源馆库存充足（> 5本且利用率低）

- **调拨数量计算**：
  ```
  调拨数量 = min(
      源馆可用库存 × 0.3,  // 最多调出源馆30%的库存
      目标馆需求缺口,       // 目标馆需要的数量
      5                     // 单次最多调拨5本
  )
  ```

- **审批流程**（混合模式）：
  - 小批量调拨（≤ 3本）：自动审批并执行
  - 大批量调拨（> 3本）：等待人工审批

- **执行时机**：
  - 定时任务：每天凌晨2点执行一次
  - 手动触发：支持管理员手动触发

### 执行流程

1. **库存分析**：
   - 查询所有图书馆的书目统计信息
   - 识别需求旺盛的馆（目标馆）
   - 识别库存充足的馆（源馆）

2. **生成调拨建议**：
   - 为每个需求馆寻找最佳供应馆
   - 计算调拨数量和优先级评分
   - 生成调拨建议并保存到数据库

3. **审批执行**：
   - 小批量调拨：自动审批并执行

### 测试场景矩阵

| 测试场景 | 前置条件 | 预期结果 | 关键验证点 |
| :--------- | :--------- | :--------- | :----------- |
| 自动审批执行 | 调拨数量≤3本 | 自动创建调拨单 | 建议状态EXECUTED，调拨单创建成功 |
| 人工审批执行 | 调拨数量>3本 | 等待审批 | 建议状态PENDING，控制台打印通知 |

| 无需调拨 | 所有馆库存均衡 | 不生成建议 | 建议列表为空 |
| 手动触发 | 任意状态 | 立即执行分析 | 控制台打印分析日志 |
