# 书籍跨馆调配 + 座位动态分配 联动改进建议

> 本文聚焦于当前「书籍跨馆调配（Agent）」与「座位动态分配（三层架构）」的联动现状，指出不足之处，并给出可选的后续打磨方向，便于你后续分阶段排期。

---

## 一、当前联动现状与不足

### 1.1 联动现状（已具备的能力）

- **数据层面的关联能力已具备**：
  - `tb_seat_reservation.borrow_id`：可将一次座位预约与一条借阅记录（书籍跨馆调配结果）关联起来；
  - `tb_seat_reservation.source`：可区分 `WALK_IN`（自主预约）与 `BORROW_PICKUP`（借书/取书推荐入口）。
- **业务流程上的解耦设计**：
  - 图书借还与跨馆调配走智能体（`UserDemandAgent` + `InventoryAgent` + `TransferService` 等）；
  - 座位查询、预约、签到、暂离、结束使用走标准三层架构（Controller + Service + Mapper）。
- **联动方式（目前可行但未完全产品化）**：
  - 借书/取书成功后，前端可以依靠返回的馆和日期信息，引导用户跳转到座位页面，携带 `borrowId` 和推荐的 `libraryId + date`，调用：
    - `GET /api/seat/available` 查询可用座位；
    - `POST /api/seat/reservation` 创建与本次借阅关联的座位预约。

从“系统能力”上看，已经**具备打通书籍调配和座位使用数据链路的基础**。

### 1.2 不足与缺口（体验与产品层面）

1. **借书/取书结果未显式返回“推荐座位参数”**  
   - 目前借书 / 取书的返回体中尚未包含：  
     - `recommendSeatLibraryId`（如取书馆ID）；  
     - `recommendSeatDate`（如当天或明天）；  
     - `actionRecommendSeat` 或类似布尔标记。  
   - 导致前端需要自己推断“在哪个馆、哪天推荐用户占座”，缺少统一规范。

2. **前端联动入口尚未规范化**  
   - 虽然后端已支持 `borrow_id + source`，但从用户视角，目前还没有一个**统一且突出的 UI 入口**：  
     - “借书成功 → 一键占座”；  
     - “取书成功 → 是否顺便预约自习位？”  
   - 这部分逻辑现在主要还在“可实现”层面，未被文档化为产品行为。

3. **座位 NO_SHOW（未签到）的信用规则尚未与整体信用体系打通**  
   - 座位侧有 `no-show-after-minutes` 配置，但目前更多是行为规则，未与 `tb_user.credit_score` 建立统一的惩戒策略；  
   - 图书预约逾期与座位未签到如果都要影响信用分，建议在文档中统一规划“信用事件模型”，否则容易各自为政。

4. **管理视角的联动信息尚不直观**  
   - 目前尚未提供如下视图：  
     - “通过借书/取书推荐产生的座位预约占比”；  
     - “某个借阅用户同时在本馆占了几个座位”的管理报表；  
     - “跨馆调配带来的到馆学习行为变化”等统计。  
   - 虽然原始数据都已经具备（`borrow_id + source + seat`），但缺少汇总接口和视图。

5. **结束使用与后续行为联动尚未定义**  
   - 现在结束使用仅更新座位、预约自身状态，并未：  
     - 触发任何“学习时长统计”；  
     - 也未与图书借还行为进一步联动（例如到馆学习时长对某些图书服务策略的影响）。

## 二、可补充 / 打磨的功能建议

以下建议按“投入小 / 收益高”到“中长期规划”排列，可作为后续迭代的候选功能列表。

### 2.1 借书/取书结果中返回座位推荐参数（强烈推荐，详细设计）

#### 2.1.1 设计目标与原则

- **目标**：在“不强绑定座位”的前提下，让借书/取书流程自然生长出一个“顺手占座”的入口。  
- **原则**：
  - 不改变当前借书/取书核心业务逻辑（不在 Agent 内直接占座）；  
  - 通过**响应体中的推荐字段** + 前端跳转，完成松耦合联动；  
  - 所有“是否展示入口”的决策由后端提供**建议字段**，前端只做简单判断。

#### 2.1.2 借书/取书结果推荐字段设计

以“借书请求结果 DTO（例如 `BorrowResult`）”和“取书确认结果 DTO（例如 `PickupResult`）”为例，建议新增或扩展如下字段（命名可根据你现有 DTO 稍作调整）：

- **推荐参数字段**：
  - `recommendSeatLibraryId` (Long)：推荐占座的馆ID  
    - 本地借书：通常就是 `preferredLibraryId` 或借书馆ID；  
    - 跨馆调拨：通常是目标馆（取书馆）ID；  
  - `recommendSeatDate` (String, yyyy-MM-dd)：推荐预约座位日期  
    - 本地借书：默认当天；  
    - 跨馆调拨取书：可设为取书成功当天或次日（可配置，如 `today`/`tomorrow`）；  
  - `recommendSeatReason` (String)：推荐原因  
    - 建议值：`LOCAL_LOAN` / `TRANSFER_PICKUP` / `RESERVATION_PICKUP` 等；  
  - `actionRecommendSeat` (Boolean)：是否建议前端展示“预约座位”按钮  
    - 例如：本地借书和到馆取书成功场景置为 `true`，部分特殊场景（如自助机借阅、短期借阅）可以设置为 `false`。

- **可选扩展**：
  - `recommendSeatTimeSlot`：推荐的时间段模板（如 `"EVENING"` 表示 18:00–22:00），供前端选默认时间段；  
  - `recommendSeatHint`：给用户看的提示语（如“建议今天在理科馆自习区预约一个座位~”）。

> 这些字段可以全部放在借书/取书结果的 `data` 部分，也可以封装成一个嵌套对象 `seatRecommend`，视你现有 DTO 风格决定。

#### 2.1.3 后端不同场景下的推荐策略

以借书结果的 `strategy` 字段为例（`LOCAL_LOAN` / `TRANSFER_PROVIDE` / `RESERVATION`）：

- **场景A：本地直接借阅（LOCAL_LOAN）**
  - `recommendSeatLibraryId = preferredLibraryId`（即借书馆）；  
  - `recommendSeatDate = 今天`；  
  - `recommendSeatReason = "LOCAL_LOAN"`；  
  - `actionRecommendSeat = true`。

- **场景B：跨馆调拨（TRANSFER_PROVIDE）**
  - 当调拨完成并可取书时（取书确认结果）：  
    - `recommendSeatLibraryId = pickupLibraryId`（取书馆）；  
    - `recommendSeatDate = 取书当日` 或 `次日`（由业务规则决定，比如你希望鼓励当天学习就用当天）；  
    - `recommendSeatReason = "TRANSFER_PICKUP"`；  
    - `actionRecommendSeat = true`。
  - 在“调拨发起但尚未到馆”的阶段，可以不返回推荐（避免时间不确定），或仅返回 `actionRecommendSeat=false`。

- **场景C：预约排队（RESERVATION）**
  - 创建预约时通常不会马上到馆，暂不推荐座位：  
    - `actionRecommendSeat = false`；  
  - 当预约兑现（图书到馆、用户可取书）时，可以类比“取书”场景，返回座位推荐参数。

#### 2.1.4 前端交互流程建议

1. **借书/取书成功页**
   - 后端返回结构中包含推荐字段：  
     ```json
     {
       "code": 200,
       "message": "借书成功，请在30天内归还",
       "data": {
         "...": "...",
         "recommendSeatLibraryId": 1,
         "recommendSeatDate": "2026-03-10",
         "recommendSeatReason": "LOCAL_LOAN",
         "actionRecommendSeat": true
       }
     }
     ```
   - 前端判断：`if (actionRecommendSeat) { 显示“预约座位”按钮 }`。

2. **点击“预约座位”按钮后的行为**
   - 跳转到座位预约页或弹出座位预约对话框，预填：
     - 馆：`libraryId = recommendSeatLibraryId`  
     - 日期：`reservationDate = recommendSeatDate`
   - 座位预约页内部流程：
     - 调 `GET /api/seat/available` 查询可用座位（可用 `autoAssign=true` 直接推荐一个）；  
     - 用户确认后，调 `POST /api/seat/reservation` 创建座位预约：
       - `libraryId = recommendSeatLibraryId`  
       - `reservationDate` / `startTime` / `endTime` 由用户或前端默认选择；  
       - `borrowId = 当前借书/取书对应的 borrowId`；  
       - `source = "BORROW_PICKUP"`（或不传，后端自动推断）。

3. **成功后提示**
   - 在座位预约成功结果中，前端可以展示如：“已为你在理科馆一楼自习区预约 18:00–22:00 的座位 A-01”。

#### 2.1.5 与现有代码的关系与改动范围

- **新增/修改点主要集中在**：
  - 借书结果 DTO（例如 `BorrowResult`）和取书结果 DTO（`PickupResult` 或相关返回对象）；  
  - `UserDemandAgent` / 借书 Controller / 取书 Controller 中组装返回结果的逻辑。
- **对现有核心流程无破坏性影响**：
  - 不改变借还书与跨馆调配的状态机；  
  - 不改变座位模块的接口与逻辑；  
  - 仅扩展返回字段，提供“推荐参数”给前端。

> 从实现成本 vs 收益的角度，这一块改动小、收益高，是非常适合作为“书籍跨馆调配 + 座位动态分配”联动的第一步落地项。 

### 2.2 统一的信用事件模型（中等优先）

**问题**：图书预约逾期与座位 NO_SHOW 现在是分离的规则，容易出现“图书严格惩罚、座位不惩罚”或相反的不一致体验。  

**建议**：

- 在文档中定义一个简单的“信用事件”表或规范，例如：
  - `EVENT_BORROW_OVERDUE`、`EVENT_RESERVE_NO_SHOW`、`EVENT_SEAT_NO_SHOW` 等；
  - 每个事件的扣分策略、封顶策略、是否会冻结账号。
- 在座位的 NO_SHOW 定时任务中：  
  - 参考图书预约逾期逻辑，将座位 NO_SHOW 也记为一次“信用事件”；  
  - 统一通过某个 `CreditService` 来更新 `tb_user.credit_score`，保证风格一致。

### 2.3 管理/报表视角的联动统计（中等优先）

**目标**：帮助管理员看到“书籍跨馆调配 + 座位使用”的整体效果。  

**可选统计接口**（示例）：

- “通过借书/取书推荐产生的座位预约统计”：
  - 统计一段时间内 `source='BORROW_PICKUP'` 的座位预约数量、占比；  
- “按馆的座位利用率和跨馆借阅用户使用情况”：
  - 例如在某馆中，有多少用户是通过跨馆调拨获得图书并在本馆占座；  
- “用户学习行为画像”：
  - 对单个用户，结合借还记录 + 座位 COMPLETED 记录，形成简单的学习时长曲线。

这些统计不会改变业务闭环，但可以为后续策略（如智能调拨权重、座位推荐逻辑）提供依据。

### 2.4 结束使用后的扩展行为（中长期）

当前结束使用仅做：

- 预约 `ACTIVE → COMPLETED`；
- 座位 `OCCUPIED → AVAILABLE`；
- 清理暂离字段。

可以考虑逐步增加：

- **记录使用时长**：  
  - 例如在 `tb_seat_reservation` 中增加 `actual_duration_minutes`，根据 `check_in_time` ~ 结束时间计算；  
- **与智能体策略的软联动**：
  - 例如：经常在某馆长时间自习的用户，下次借书时优先推荐该馆的调拨或本地借阅策略。

## 三、短期推荐落地顺序

若你希望对“书籍跨馆调配 + 座位动态分配”的结合度再提升一小步，而不大动架构，建议按如下顺序迭代：

1. **小步快跑**：  
   - 在借书/取书结果中增加 `recommendSeatLibraryId` / `recommendSeatDate` / `actionRecommendSeat` 等字段；  
   - 在前端实现“借书成功 → 一键预约座位”的入口；  
   - 在座位预约接口中带上 `borrowId` + `source='BORROW_PICKUP'`。

2. **中期**：  
   - 将座位 NO_SHOW 与信用体系打通，文档化“信用事件模型”；  
   - 补充一两个关键统计接口（如“借书推荐带来的座位预约占比”）。

3. **长期**：  
   - 结合学习时长、借阅习惯、馆内座位利用率，考虑是否需要一个真正的“座位推荐智能体”，在当前三层逻辑之上做更高层次的策略优化。

以上建议都基于你当前的实现，不要求引入新的中间件，也不会破坏现有闭环，只是在已有能力之上按“产品视角”进行打磨与增强。
