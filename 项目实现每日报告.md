## 20260225-智能借书与跨馆调拨模块开发总结

### 一、主要工作内容

本阶段完成了图书馆多智能体调度系统的核心功能——**智能借书与资源调拨**的完整实现。具体包括：

**1. 数据模型重构与基础搭建** 将原有的简单库存记录改造为**单本追踪体系**（每册书拥有唯一条码号），实现了图书副本级别的精细化管理。完成了13张业务表的实体映射与基础CRUD能力构建，为跨馆调拨提供了数据支撑。

**2. 多智能体协商机制的实现** 基于FIPA Contract Net协议构建了完整的协商流程：

- **UserDemandAgent**（需求代理）作为协调者，负责任务分发与决策；
- **InventoryAgent**（库存代理）作为参与者，基于本地库存、距离成本、服务压力三维度计算投标评分；
- 实现了"广播-投标-评标-履约"的完整协商闭环，平均协商耗时控制在3秒以内。

**3. 三种业务策略的覆盖** 系统根据实时库存状态自动选择最优策略：

- **本地借阅**：目标馆有库存时直接办理；
- **跨馆调拨**：目标馆缺货但其他馆有库存时，自动触发跨馆运输并预留24小时取书期；
- **预约排队**：全馆缺货时创建预约记录，书归还后自动优先分配。

**4. 业务规则引擎** 集成了借阅权限检查（最大借阅数限制、重复借书拦截）、库存同步机制（调拨时实时更新两端统计表）、以及状态流转控制（AVAILABLE→IN_TRANSIT→RESERVED→BORROWED）。

### 二、关键技术问题与解决方案

**问题1：Spring容器与JADE Agent启动时序冲突** 

现象：Agent启动时尝试获取Spring Bean，因上下文未初始化完成导致空指针异常。 

解决：采用延迟加载策略，通过实现`ApplicationRunner`接口确保Spring全量启动后再初始化Agent容器，并增加1秒缓冲期保证依赖注入完成。

**问题2：调拨状态死锁** 

现象：首次调拨成功后，再次查询该书显示库存为0，无法二次调拨。 根因：调拨完成后副本状态停留在"运输中"（IN_TRANSIT），未被重置为可借状态。 

解决：优化状态机设计，采用"即时到达"模式——调拨完成后立即将副本置为AVAILABLE（或根据业务场景置为RESERVED），避免状态悬空。

**问题3：库存统计不一致** 

现象：物理调拨已完成（tb_book_copy.library_id已变更），但统计表（tb_library_biblio_stats）仍显示原馆有库存、目标馆无库存。

解决：在调拨事务中强制增加统计表的双向更新（源馆库存-1，目标馆库存+1），通过数据库事务保证两份数据的一致性。

**问题4：预约功能缺失** 

现象：初期设计的预约表（tb_book_reservation）未接入业务流，全馆缺货时系统直接返回错误而非创建预约。

解决：扩展协商逻辑，当所有InventoryAgent均返回REFUSE时，自动创建PENDING状态的预约记录，并设置3天有效期，实现无库存场景的兜底处理。

### 三、实现成果与验证

目前该模块已通过全流程测试验证：

- **功能完整性**：覆盖了"有书直借、无书调拨、全馆无书预约"三种完整场景；
- **数据一致性**：调拨过程中tb_book_copy、tb_book_transfer、tb_library_biblio_stats三张表数据联动正确；
- **协商有效性**：通过Swagger接口可观测到完整的CFP（Call For Proposal）协商日志，包括各馆投标评分、最优方案选择、调拨执行等关键节点。

系统现已具备**从用户请求到物理库存变更的端到端闭环能力**，可作为多智能体协作调度研究的核心实验平台。

### 四、后续可扩展方向

虽然当前借书流程已完整闭环，但若要构建更完善的图书馆生态系统，建议后续补充：

- **还书时的预约自动兑现**：当用户归还图书时，系统自动检测预约队列并触发向等待者的调拨；（已完成）
- **取书确认机制**：补充从"预留"到"正式借阅"的状态转换接口；（已完成）
- **定时任务调度**：实现超期预留（24小时）和过期预约（3天）的自动释放。

以上功能可在现有架构基础上平滑扩展，无需改动已验证的核心协商机制。

## 20260226-还书自动分配与调拨履约闭环开发总结

### 一、主要工作内容

本阶段完成了图书馆资源调度系统的**还书流转与调拨履约**核心闭环，实现了从"用户还书"到"预约者取书"的全自动资源再分配。具体包括：

**1. 还书自动分配引擎（BookReturnService）**

重构了还书流程，实现了"归还即分配"的智能调度：

- **队列遍历机制**：采用FIFO策略检查预约队列，自动跳过已达借阅上限的无效预约者，确保资源分配给有效用户；
- **双模式履约**：支持"就地预留"（同馆还书直接锁定）和"跨馆调拨"（异馆还书触发物流运输）两种场景；
- **信用评估体系**：集成逾期检测逻辑，按"每天2分、上限20分"规则自动扣除信用分，低于60分自动冻结账号。

**2. 调拨履约闭环（TransferService）**

构建了独立的调拨完成回调体系，解决了物流"最后一公里"问题：

- **状态机完善**：引入`IN_TRANSIT`（运输中）状态，填补`AVAILABLE`到`RESERVED`之间的状态真空；
- **双源兼容设计**：智能识别"借书调拨"（更新已有Borrow记录）与"还书调拨"（新建Borrow记录）两种业务来源，统一走`/api/transfer/complete`接口完成闭环；
- **默认位置管理**：调拨完成后自动设置"馆X-调拨暂存区"默认位置，等待馆员后续精确上架。

**3. 防重与并发控制机制**

堵塞了恶意重复申请的漏洞：

- **TRANSFERRING状态**：创新性地利用Borrow表记录运输中状态，在发起调拨时即插入记录（状态TRANSFERRING），而非等到货后才创建；
- **全链路防重**：在借书入口增加"活跃借阅检查"（TRANSFERRING/RESERVED/BORROWING任一状态均拦截），杜绝了同一用户重复调拨同一本书的可能；
- **库存锁定**：运输中的副本状态锁定为`IN_TRANSIT`，防止被其他用户借走。

**4. 架构清晰化调整**

优化了系统边界与职责划分：

- **人工确认机制**：移除调拨的自动定时完成逻辑，改为必须由物流人员/馆员手动调用回调接口确认收货，符合真实物流场景；
- **常量集中管理**：将散落各处的状态字符串（IN_TRANSIT, TRANSFERRING等）统一收拢至`LibraryConstants`，消除硬编码风险；
- **Controller规范化**：将手动拼接JSON的不规范做法改为直接返回Java对象，利用Spring自动序列化，提升接口稳定性。

### 二、关键技术问题与解决方案

**问题1：Borrow表非空约束冲突**

现象：创建运输中记录时抛出`Field 'due_time' doesn't have a default value`异常。
根因：MyBatis插入时未设置`due_time`字段，而数据库列约束为NOT NULL。
解决：在创建TRANSFERRING状态记录时，将`due_time`初始化为当前时间（作为占位），待调拨完成更新为RESERVED时再重置为"当前时间+24小时"的真实过期时间。

**问题2：调拨完成时的记录查找失败**

现象：调用`/api/transfer/complete`返回"未找到关联的预约记录"。
根因：早期设计假设调拨完成必须通过`reservation`表查找用户，但借书调拨流程中并未创建预约记录（直接创建了Borrow记录）。
解决：重构`completeTransfer`方法，采用"双源探测"策略：优先查询`tb_book_borrow`（TRANSFERRING状态），找不到再回退查询`tb_book_reservation`（FULFILLED状态），并根据查找结果分支处理（更新 vs 插入）。

**问题3：状态值与数据库枚举不匹配**

现象：副本状态被设置为`IN_TRANSIT`，但数据库实际报错或查询异常。
根因：`tb_book_copy.status`字段的COMMENT仅列出AVAILABLE/BORROWED/RESERVED/LOST/DAMAGED，未包含IN_TRANSIT。
解决：修改表结构，将status字段扩展为支持6种状态，明确`IN_TRANSIT`表示"调拨运输中"的中间态。

**问题4：重复调拨漏洞**

现象：同一用户可无限次发起同一本书的调拨申请，直到所有副本被占满。
根因：仅检查`reservation`表是否存在记录，但借书调拨不走预约表。
解决：建立"活跃借阅"统一检查机制，在借书入口查询`tb_book_borrow`中是否存在该用户的TRANSFERRING/RESERVED/BORROWING记录，实现全链路防重。

### 三、实现成果与验证

目前该模块已通过边界条件测试与异常流验证：

- **防重有效性**：同一用户连续两次申请同一本书，第二次被拦截并提示"您已借阅该书（可能在调拨中）"；
- **状态流转准确性**：发起调拨→副本IN_TRANSIT/Borrow-TRANSFERRING→完成回调→副本RESERVED/Borrow-RESERVED（24小时倒计时启动），状态转换完整可追溯；
- **双场景覆盖**：借书调拨（用户A从馆1调书到馆2）与还书调拨（用户B还书到馆1，触发向馆3的预约者调拨）均能通过同一回调接口正确完成，用户计数更新准确（借书调拨+0，还书调拨+1）；
- **人工确认机制**：调拨单创建后保持IN_TRANSIT状态30分钟以上，直至手动调用complete接口才状态翻转，符合"物理运输需要人工确认"的业务现实。

系统现已具备**借书-运输-到货-预留-取书**与**还书-检测-再分配**的双向闭环能力，资源流转效率达到设计预期。

### 四、后续可扩展方向

虽然当前调拨履约与还书分配已完整闭环，但若要构建无人值守的智慧图书馆，建议后续补充：

- **超期释放定时任务**：开发`ReservationMonitorJob`，每小时扫描`tb_book_borrow`中已过期的RESERVED记录（24小时未取）和`tb_book_reservation`中过期的PENDING记录（3天未到货），自动释放资源给队列中的下一位用户；
- **取书确认接口**：补充`confirmPickup`接口，将RESERVED状态的Borrow记录转为正式的BORROWING状态，并精确记录取书时间；
- **座位预约系统**：基于已构建的`tb_seat`与`tb_seat_reservation`表，实现与图书调拨联动的"到馆取书+座位预留"一体化服务；
- **动态库存平衡Agent**：基于`tb_library_biblio_stats`的借阅热度分析，自动触发低利用率馆向高需求馆的主动调拨（而非被动响应）。

### 二、关键技术问题与解决方案

**问题1：Result.success()方法调用错误**

现象：`TransferSuggestionController`中调用`Result.success()`只传String参数，编译报错。
根因：方法签名要求`T data`或`(String message, T data)`，不能只传String。
解决：修改所有调用处，传入null作为data参数：`Result.success("审批通过", null)`。

**问题2：调拨原因字段未正确设置**

现象：创建调拨单时，`transfer_reason`字段始终为默认值`USER_REQUEST`，而非`INVENTORY_BALANCE`。
根因：`TransferService.createTransfer()`方法未支持指定调拨原因。
解决：添加重载的`createTransfer`方法，支持传入`transferReason`和`suggestionId`参数，保持原有方法的向后兼容性。

**问题3：调拨建议可能重复生成**

现象：每次执行库存平衡分析时，可能对同一本书生成多条相同的调拨建议。
根因：未检查是否已存在待审批的建议。
解决：在`analyzeBiblio`方法中添加去重检查，查询是否已存在相同的PENDING状态建议，存在则跳过。

**问题4：审批流程状态错误**

现象：审批通过后，建议状态变为`APPROVED`，但调用`executeSuggestion`时报错"调拨建议状态不正确"。
根因：`executeSuggestion`方法只允许`PENDING`状态执行，但审批后状态已变为`APPROVED`。
解决：修改`executeSuggestion`方法，允许`PENDING`或`APPROVED`状态执行，支持两种状态流转路径：
- 自动审批：PENDING → EXECUTED
- 人工审批：PENDING → APPROVED → EXECUTED

```java
private Integer calculateProgressPercentage(BookTransfer transfer) {
    String status = transfer.getStatus();
    if ("PENDING".equals(status)) {
        return 0;
    } else if ("COMPLETED".equals(status)) {
        return 100;
    } else if ("CANCELED".equals(status)) {
        return 0;
    } else if ("IN_TRANSIT".equals(status)) {
        LocalDateTime requestTime = transfer.getRequestTime();
        LocalDateTime estimatedArrivalTime = transfer.getEstimatedArrivalTime();
        LocalDateTime now = LocalDateTime.now();
        
        Duration totalDuration = Duration.between(requestTime, estimatedArrivalTime);
        Duration usedDuration = Duration.between(requestTime, now);
        
        long totalSeconds = totalDuration.getSeconds();
        long usedSeconds = usedDuration.getSeconds();
        
        if (totalSeconds <= 0) {
            return 50;
        }
        
        int percentage = (int) ((usedSeconds * 100) / totalSeconds);
        return Math.min(percentage, 95);
    }
    return 0;
}
```

```java
private String formatDuration(long minutes) {
    if (minutes < 60) {
        return minutes + "分钟";
    } else if (minutes < 24 * 60) {
        long hours = minutes / 60;
        long mins = minutes % 60;
        return hours + "小时" + (mins > 0 ? mins + "分钟" : "");
    } else {

}
```

#### 1.4 核心功能实现

**功能A2：预留超期释放闭环**

- 定时任务每小时扫描超期的RESERVED借阅记录
- 自动释放预留资源：
  - 借阅状态：RESERVED → CANCELLED
  - 副本状态：RESERVED → AVAILABLE
  - 用户计数：current_borrow_count - 1
  - 信用分：credit_score - 5（低于60分冻结账号）
- 自动兑现下一位预约者：
  - 查找PENDING状态的预约记录
  - 按reserve_time排序，优先兑现最早预约者
  - 创建新的RESERVED借阅记录或发起调拨

- 支持分页查询：page, size参数

### 二、调拨完成接口优化

#### 2.1 优化内容

**解决方案**：
- 移除 `actualArriveTime` 参数
- 系统自动使用 `LocalDateTime.now()` 作为完成时间
- 简化接口调用，降低调试成本

#### 2.2 改动文件

- `TransferController.java`：
  - 移除 `TransferCompleteRequest.actualArriveTime` 字段
  - 移除 `BatchCompleteRequest.actualArriveTime` 字段
  - 移除 `LocalDateTime` 导入

- `TransferService.java`：
  - `completeTransfer` 方法移除 `actualArriveTime` 参数
  - `completeBatchTransfer` 方法移除 `actualArriveTime` 参数
  - 直接使用 `LocalDateTime.now()` 获取当前时间

#### 2.3 优化效果

**优化前**：
```json
{
  "transferId": 20,
  "actualArriveTime": "2024-01-15T14:30:00"
}
```

**优化后**：
```json
{
  "transferId": 20
}
```

**优势**：
- 简化接口调用
- 避免时间格式错误
- 避免时区问题
- 降低调试成本
