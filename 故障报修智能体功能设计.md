## 一、背景与总体目标

在当前系统中，已经完成了以下三类智能体及其配套功能：

1. **用户需求感知智能体（UserDemandAgent）**：负责感知用户借阅需求、触发智能借书与跨馆调拨策略，并在本地借阅场景中给出座位推荐。  
2. **书籍库存智能体（InventoryAgent / 调拨相关 Agent）**：负责基于各馆库存与调拨成本完成跨馆调拨决策与执行。  
3. **座位预约智能体（Seat 相关服务 + 控制器）**：负责座位可用性查询、动态分配、预约创建/取消、签到与暂离等。

从导师给出的参考 “① 划分智能体角色（用户需求感知智能体、书籍库存智能体、座位预约智能体、故障报修智能体）；② 设计多智能体通信协议，解决 ‘书籍跨馆调配 - 座位动态分配’ 协同问题” 来看，当前系统中**尚缺的就是“故障报修智能体”这一块能力**，主要用于：

- 统一承接“图书馆设施/资源的故障与报修请求”，形成结构化工单；  
- 将**资源状态（图书副本、座位、区域、馆内设备）**与“可用性/调度决策”联动，例如：  
  - 某个座位损坏 → 自动从座位可用性候选中剔除；  
  - 某馆部分区域施工/停用 → 降低或暂停该区域的座位/藏书分配；  
  - RFID 门禁 / 借还设备异常 → 影响借书/还书流程时给出友好提示；
- 通过与现有智能体的通信协议，**在“书籍跨馆调配 – 座位动态分配”协同时，考虑资源健康状况**，避免把用户分配到“坏座位”“维修中馆区”。

本设计文档聚焦于：**故障报修智能体的职责划分、数据模型、核心业务流程、多智能体通信协议以及与现有模块的集成方式**，暂不涉及具体代码实现。

## 二、角色定位与边界划分

### 2.1 故障报修智能体（FaultRepairAgent）的核心职责

1. **故障感知与受理**  
   - 接收来自前端的“用户报障”请求（如：座位损坏、空调异常、电源插座无法使用、机器故障等）；  
   - 接收来自系统监控/后台任务的自动告警（如：调拨任务反复失败、座位占用状态与实际使用检测不一致等）；  
   - 将上述信息统一转化为“故障工单”，写入故障表。

2. **状态管理与调度建议**  
   - 维护故障的生命周期：**存库使用英文状态码**（`REPORTED` → `ACCEPTED` → `IN_PROGRESS` → `RESTORED` / `CLOSED`），界面展示用中文映射；  
   - 根据故障对象（座位/座位区域/图书馆/设备）**更新对应资源的可用状态**，例如：  
     - 座位损坏 → 标记该座位为 UNAVAILABLE / MAINTENANCE，Seat 模块不再分配该座位；  
     - 区域施工 → 降低该区域在座位推荐中的权重，或直接暂停分配；  
     - 馆级设备严重故障 → 暂停向该馆分配跨馆调拨或座位推荐。  

3. **与其他智能体的协同**  
   - 与 UserDemandAgent 协同：在出借/调拨/座位推荐时，提供“馆级/区域级健康状况”反馈，必要时建议切换候选馆或不推荐受影响区域；  
   - 与 InventoryAgent 协同：当某馆因故障暂时不可用（如封馆维护），调拨智能体应临时降低或关闭向该馆的调拨；  
   - 与座位模块协同：对损坏座位和关闭区域进行下线或降权，避免系统继续将用户分配到异常资源。

4. **对运维/管理员的支撑**  
   - 提供运维视角的**故障列表查询接口**（按馆/区域/类型/状态筛选）；  
   - 支持管理员手动更新工单状态、补充维修记录和备注；  
   - 预留与外部工单系统（如校内统一报修平台）对接的扩展点。

### 2.2 与其他智能体的分工关系

- **UserDemandAgent**：关注“用户需求决策”（借书策略、座位推荐），不直接管理资源健康，只在决策时向 FaultRepairAgent 查询“资源是否可用/是否处于故障中”。  
- **InventoryAgent**：关注“书籍库存与调拨决策”，在选择目标馆时可以参考 FaultRepairAgent 提供的“馆级可用性/负载状态”。  
- **Seat 模块（SeatReservationService + SeatAvailabilityService 等）**：继续承担“座位业务流程”，但在座位筛选/可用性判断时需要**订阅或查询**故障信息。  
- **FaultRepairAgent**：不直接发起借书/调拨/预约，只负责“资源健康状态”的管理和告知，是一个横切的“资源健康治理层”。

## 三、数据模型与配置建议

本节先说明**故障类型、严重程度的用途与取值**，再给出设备表、工单表及可选字典表的设计；最后说明与现有资源表的关系及配置规则。

### 3.1 故障类型与严重程度的用途与取值

**（1）故障类型 fault_type**

用于描述“什么坏了”，便于前端展示、运维分工和规则配置。建议**少量、可读、语义明确**，避免抽象英文码。用途包括：

- 前端报修时用户选择、列表筛选与统计；  
- 运维判断处理优先级或分配负责人；  
- 业务规则中可按类型决定“是否参与自动下线资源”（见下）。

建议取值（存库用短码，界面展示用中文）：

| 存储值（code） | 展示名     | 说明与典型用途 |
|----------------|------------|----------------|
| seat_broken    | 座椅损坏   | 座位不可用，参与“排除该座位”的规则 |
| power_failure  | 电源故障   | 插座/供电异常，可排除座位或区域 |
| env_issue      | 环境问题   | 冷热、噪音等，可按严重程度降权或排除 |
| network_fault  | 网络故障   | 馆/区域网络异常，可标注馆或区域不可用 |
| other          | 其他       | 由严重程度决定是否排除资源 |

后续若需扩展（如“门禁故障”“空调故障”），可在上述基础上增加，或与设备类型联动。

**（2）严重程度 severity**

用于决定**该故障对资源可用性的影响力度**，避免“所有状态一视同仁、不知道有什么用”。建议**三级即可**，用途明确：

- **低**：仅记录，不自动从推荐中排除该资源，不主动推送给运维；  
- **中**：从推荐中排除该资源（座位/区域/馆），并可选通知运维；  
- **高**：立即排除并通知运维，必要时通知已预约该资源的用户。

建议取值：

| 存储值 | 展示名 | 说明 |
|--------|--------|------|
| low    | 低     | 仅记录，不参与“不可用”判断 |
| medium | 中     | 参与排除，座位/区域/馆在推荐时被过滤 |
| high   | 高     | 同“中”，且优先通知与统计 |

实现时可在配置文件中按 `severity` 控制“哪些程度参与排除”“哪些程度触发通知”，无需在表里再堆更多状态。

---

### 3.2 设备表：是否必要、如何设计

**是否需要设备表**

- **仅做“座位 / 区域 / 馆”报修**：可以不建设备表。工单表直接使用 `seat_id`、`area_id`、`library_id` 关联现有业务表即可，报修对象类型用“座位 / 区域 / 馆”即可区分。  
- **需要支持“馆内设备”报修**（如门禁、空调、借还书机、网络节点等）：这些对象在系统里没有单独的 `tb_seat`/`tb_seat_area`，建议新增**设备表**（本项目中落地为 **`tb_equipment`**），工单通过 **`equipment_id`** 关联；座位/区域/馆仍用工单上的 `seat_id`/`area_id`/`library_id` 关联，不强制在设备表里为每个座位建一条记录。

**推荐做法**：增加一张**设备表**，只存放“馆内设备”（非座位、非区域、非馆本身），工单表同时支持“直接关联座位/区域/馆”和“关联设备”。这样：

- 座位、区域、馆继续用现有表，工单直接外键关联；  
- 门禁、空调等有明确归属（馆/区域）和名称，便于报修与统计；  
- 扩展时只需在设备表增加类型或记录，无需改工单表结构。

**设备表 tb_device（馆内设备）**

仅表示“可被报修的馆内设备”，与 `tb_seat`、`tb_seat_area`、`tb_library` 并列，不替代业务表。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| library_id | bigint | 所属馆，非空 |
| area_id | bigint | 所属区域，可空 |
| name | varchar(100) | 设备名称，如“一楼东侧空调”“东门门禁” |
| device_type | varchar(20) | 设备类型：门禁 / 空调 / 借还书机 / 网络 / 其他 |
| status | varchar(20) | 设备状态：正常 / 故障 / 停用（可与工单联动更新） |
| remark | varchar(200) | 备注，可空 |
| create_time | datetime | |
| update_time | datetime | |

- 门禁、空调、借还书机、网络等在此建记录；报修时选“某馆某设备”，工单填 `equipment_id`（对应 `tb_equipment`）。  
- 座位、区域、馆**不在此表建记录**，报修时工单填 `seat_id`/`area_id`/`library_id`，`equipment_id` 为空。

### 3.3 故障工单表 tb_fault_report

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| library_id | bigint | 所属馆，可空（系统级或未知时为空） |
| area_id | bigint | 所属区域，可空 |
| seat_id | bigint | 所属座位，可空 |
| equipment_id | bigint | 关联馆内设备（`tb_equipment.id`），可空（报修设备时填写；与设备表命名一致） |
| fault_type | varchar(30) | 故障类型：见 3.1 表（如 seat_broken / power_failure / env_issue / network_fault / other） |
| severity | varchar(20) | 严重程度：low / medium / high |
| status | varchar(20) | 工单状态：**存库英文码**，见下表；默认 `REPORTED` |
| title | varchar(200) | 简要标题 |
| description | text | 报修人填写的详细描述，可空 |
| admin_remark | varchar(500) | 运维/管理员处理备注（与 `description` 分离），可空 |
| report_source | varchar(20) | 报修来源：用户 / 监控 / 管理员 / 系统（亦可约定英文码，与实现统一即可） |
| report_user_id | bigint | 报修用户ID，可空 |
| assignee | varchar(50) | 当前负责人，可空 |
| created_time | datetime | |
| updated_time | datetime | |
| resolved_time | datetime | 恢复/关闭时间，可空 |

**工单状态 status（存库英文码 ↔ 界面展示）**：

| 存库值 | 展示名 |
|--------|--------|
| REPORTED | 已报修 |
| ACCEPTED | 已受理 |
| IN_PROGRESS | 维修中 |
| RESTORED | 已恢复 |
| CLOSED | 已关闭 |

**约束与约定**：

- 至少填写一类对象：`seat_id`、`area_id`、`library_id`、`equipment_id` 至少一个非空，便于“按资源维度”查询故障和参与排除逻辑。  
- `fault_type`、`severity` 使用 3.1 的取值，便于规则引擎和前端展示（前端可用字典或枚举转中文）。  
- `status` 与配置 `active-statuses` 配合：处于 `REPORTED` / `ACCEPTED` / `IN_PROGRESS` 且 `severity` 为 medium 或 high 时参与“资源不可用”判断（具体以配置为准）；`RESTORED` / `CLOSED` 不再参与。  
- **首版不支持上传故障图片**；若后续需要，可再增加 `images` 字段（与旧表 `tb_repair_ticket` 类似）。

### 3.4 其他表（可选）

- **故障类型字典表 tb_fault_type**（可选）：若希望故障类型可配置、可扩展，可建表 `(code, name, sort_order, affect_resource)`，工单存 `code`，前端与规则读 `name` 和是否参与资源排除。当前阶段用 3.1 的固定枚举即可。  
- **严重程度字典表**：一般不需要单独表，用配置或枚举即可。  
- **工单操作日志表 tb_fault_action_log**（可选）：记录每次状态变更、处理备注与操作人，便于追溯；首版可省略，使用 `tb_fault_report` 的 `updated_time`、`resolved_time`、`assignee`、`admin_remark` 做简单跟踪。

---

### 3.5 与现有资源表的关系

- **座位 / 区域 / 馆**：工单通过 `seat_id`、`area_id`、`library_id` 与 `tb_seat`、`tb_seat_area`、`tb_library` 关联；故障状态**不直接写回业务表**，通过“查询未关闭且 severity 为 中/高 的工单”判断资源是否暂不可用。  
- **馆内设备**：通过 `equipment_id` 关联 `tb_equipment`；设备表可单独维护 `status`（正常/故障/停用），与工单状态联动（可选）。  
- 座位可用性、调拨目标馆选择等逻辑：通过联表查询或缓存“存在未关闭且中/高严重度工单的资源ID集合”，在推荐与分配时排除这些资源。

### 3.6 可配置的业务规则

在 `application-business-rules.yml`（或等价配置）中，用**可读的配置**控制“哪些工单状态、哪些严重程度”参与排除，避免在代码里写死：

```yaml
library:
  business-rules:
    fault:
      # 参与“资源不可用”判断的工单状态（须与 tb_fault_report.status 存库英文码一致）
      active-statuses:
        - REPORTED
        - ACCEPTED
        - IN_PROGRESS
      # 参与“资源不可用”判断的严重程度（未关闭且在该列表内则排除）
      exclude-severities:
        - medium
        - high
      # 馆级：仅当存在“高”严重度未关闭工单时，该馆不参与调拨/推荐
      library-exclude-only-high: true
```

### 3.6.1 馆级与座位/区域/设备：排除规则差异（实现必读）

配置项 **`library-exclude-only-high`** 仅作用于 **`resourceType = LIBRARY`** 的健康判断：

- **`LIBRARY`**：当 `library-exclude-only-high: true` 时，仅当该馆存在未关闭工单且 **`severity = high`** 且 `status ∈ active-statuses` 时，该馆视为不可用；**medium 严重度的馆级工单不导致整馆排除**。  
- **`SEAT`、`SEAT_AREA`、`DEVICE`（或 EQUIPMENT）**：仍按 **`exclude-severities`**（如 medium + high）与 **`active-statuses`** 判断；**不受** `library-exclude-only-high` 收窄。  

这样可在「全馆封馆」类场景用高严重度控制整馆，而「单座、单区、单设备」的中等严重度仍可及时从推荐中剔除。

### 3.7 配置与代码绑定（实现约定）

故障相关 YAML **必须**与 Spring 配置类同步维护，避免“文档有配置、运行读不到”：

- 在 **`application-business-rules.yml`**（或合并进 `application.yml`）中增加 `library.business-rules.fault` 段（见 3.6、第八章 8.4）。  
- 在 **`BusinessRulesProperties`** 中增加对应的嵌套类（如 `FaultRules`），字段与 YAML 键一致（如 `activeStatuses`、`excludeSeverities`、`libraryExcludeOnlyHigh`），供 `FaultService`、资源健康查询、座位/调拨联动统一注入使用。  
- 项目已在 **`application.yml`** 中通过 `spring.config.import: optional:classpath:application-business-rules.yml` 合并该文件，确保运行时能加载 `fault` 等规则；若改为仅使用 `application.yml`，需将 `fault` 段手工并入主配置。

完整示例见项目内 `application-business-rules.yml` 与 `BusinessRulesProperties.FaultRules`。

这样，故障类型和严重程度在表里语义清晰、数量可控，用途也集中在“展示 + 筛选 + 是否参与排除 + 是否通知”，便于实现和后续扩展。

## 四、核心业务流程设计

### 4.1 用户报修座位/区域故障流程

1. 用户在前端点击“座位有问题/报修”按钮，选择：  
   - 目标座位（或区域/馆）；  
   - 故障类型（如座椅损坏、电源不可用、噪音环境等）；  
   - 简单文字描述。**首版不支持上传故障图片**（后续版本可扩展字段）。  
2. 前端调用 **`POST /api/fault/report`**（见第八章 8.1，`FaultController`），写入 `tb_fault_report`，`status` 置为 **`REPORTED`**（界面展示为「已报修」）。  
3. **FaultRepairAgent** 订阅到新工单，进行初步处理：
   - 根据故障类型设定严重程度与默认策略；
   - 若严重程度为「中」或「高」，向座位模块发送“资源状态变更”通知，提示该座位暂时不要参与推荐；
   - 向运维/管理员发送通知（邮件/站内信）提醒有新报修。
4. 后续由管理员或自动脚本更新工单状态（`ACCEPTED` → `IN_PROGRESS` → `RESTORED` / `CLOSED`），并通过 FaultRepairAgent 反向通知座位模块恢复可用性。

### 4.2 系统自动感知故障流程（示例）

1. 定时任务或监控系统检测到异常：  
   - 某馆的调拨失败率在一段时间内异常升高；  
   - 某区域的座位占用率长期为 0，但预约/签到记录异常；  
2. 监控模块向 FaultRepairAgent 发送“系统发现潜在故障”的消息；  
3. FaultRepairAgent 自动创建一条 MONITOR 来源的 `tb_fault_report`，并根据策略：  
   - 降低该馆或区域在推荐/调拨中的优先级；  
   - 记录原因，等待人工确认；  
4. 管理员在后台确认后，可以将工单标记为 **`RESTORED`** 或 **`CLOSED`**（界面展示为「已恢复」「已关闭」），并触发恢复策略。

### 4.3 故障对“书籍跨馆调拨 – 座位动态分配”协同的影响

1. **书籍侧影响**  
   - 若某馆存在未关闭且严重程度为「高」的故障（例如封馆维护）：  
     - InventoryAgent 在参与调拨决策时，不再将该馆作为**目标馆**；  
     - 也可以在必要时减少/暂停将书从其他馆调入该馆。  

2. **座位侧影响**  
   - 若某座位存在未解决故障：SeatAvailabilityService 在筛选座位时直接排除该座位；  
   - 若某区域存在高严重度故障：可以整体不再返回该区域的座位，或让自动分配优先避开该区域。  

3. **跨模块综合场景**  
   - 场景示例：  
     - 用户借书成功，被推荐到 A 馆某区域某座位；  
     - 该区域突然施工，管理员或用户报修；  
     - FaultRepairAgent 收到后：  
       - 调整该区域/座位的可用性；  
       - 下一次 UserDemandAgent 或前端发起座位推荐时，系统会自动绕开该区域；  
       - 如有需要，可向已预约/已被推荐到该区域的用户发送通知，建议重新预约。

---

## 五、多智能体通信协议设计（概要）

采用 JADE ACL 消息 + 内部 DTO 的组合形式，重点说明与其他三类智能体之间的交互消息。

### 5.1 消息类型定义（概念层面）

1. `FAULT_REPORT`：新故障报修/监控告警通知  
2. `FAULT_STATUS_UPDATE`：故障状态变更（已恢复/已关闭等）  
3. `RESOURCE_HEALTH_QUERY`：查询某资源（馆/区域/座位）的健康状态  
4. `RESOURCE_HEALTH_RESPONSE`：上述查询的响应结果  
5. `RESOURCE_IMPACT_NOTIFICATION`：资源状态发生变化，对座位/调拨有影响时的主动通知

### 5.2 示例：资源健康查询协议

- **发起方**：UserDemandAgent / InventoryAgent / Seat 模块中的代理包装层  
- **目的**：在做出“调拨目标馆选择”“座位推荐”等决策前，查询相关资源是否存在未关闭的故障。  

请求内容（逻辑字段）：

```json
{
  "type": "RESOURCE_HEALTH_QUERY",
  "resources": [
    { "resourceType": "LIBRARY", "resourceId": 1 },
    { "resourceType": "SEAT_AREA", "resourceId": 10 },
    { "resourceType": "SEAT", "resourceId": 1001 }
  ]
}
```

响应内容（逻辑字段）：

```json
{
  "type": "RESOURCE_HEALTH_RESPONSE",
  "results": [
    {
      "resourceType": "LIBRARY",
      "resourceId": 1,
      "available": true,
      "hasCriticalFault": false,
      "faultSummary": null
    },
    {
      "resourceType": "SEAT",
      "resourceId": 1001,
      "available": false,
      "hasCriticalFault": true,
      "faultSummary": "座位损坏，等待维修"
    }
  ]
}
```

在实现上，可以在 JADE 中定义对应的消息封装 DTO，由 FaultRepairAgent 负责将查询拆分为对 `tb_fault_report` 的检索，并返回聚合结果。

### 5.3 示例：资源影响通知协议

- **发起方**：FaultRepairAgent  
- **目的**：当故障状态发生关键变更（如 `REPORTED`→`IN_PROGRESS`、`RESTORED`/`CLOSED`）时，主动通知相关智能体刷新本地缓存或重新计算推荐结果。

通知内容（逻辑字段）：

```json
{
  "type": "RESOURCE_IMPACT_NOTIFICATION",
  "event": "SEAT_UNAVAILABLE",
  "resourceType": "SEAT",
  "resourceId": 1001,
  "severity": "medium",
  "faultId": 123,
  "newStatus": "IN_PROGRESS"
}
```

座位模块收到后，可以：  
- 将该座位从当前缓存的可用列表中移除；  
- 若当前有进行中的推荐/预约流程，必要时给用户提示“该座位刚刚进入维修状态，请重新选择”。

---

## 六、在现有系统中的落地形态建议

综合当前项目的实现方式，建议故障报修智能体的落地形态如下：

1. **新增 JADE Agent：`FaultRepairAgent`**  
   - 类似 `UserDemandAgent`，在系统启动时由 Spring + JADE 集成代码一并创建；  
   - 负责接收与发送与故障相关的 ACL 消息，统一访问故障领域的 Service/Mapper。

2. **新增领域 Service 与 Controller**  
   - `FaultService`：封装对 `tb_fault_report` 的增删改查与状态流转逻辑；  
   - `FaultController`：对外提供 REST API（报修、查看工单、更新状态），便于前端和运维端直接调用。  
   - 通过类似 `AgentTaskManager` 的方式，将部分操作包装为“异步任务”，由 FaultRepairAgent 处理更复杂的策略。

3. **在现有模块中增加适配层**  
   - 在座位可用性查询与动态分配逻辑中，引入“故障过滤/降权”步骤，可以通过：  
     - 直接查询 `tb_fault_report`；或  
     - 通过一个“资源健康缓存服务”，由 FaultRepairAgent 定期刷新。  
   - 在跨馆调拨决策逻辑中，为目标馆选择增加“馆级健康评分”的考虑，用于避免把用户调配到短期内无法正常服务的馆。

4. **渐进式接入策略**  
   - 第一阶段：只做“座位/区域/馆的手动报修 + 手动关闭”，主要用于座位损坏、区域施工场景；  
   - 第二阶段：逐步接入监控数据（如调拨失败率、座位使用异常），由系统自动创建低/中严重度工单；  
   - 第三阶段：与通知系统、外部工单系统对接，实现更丰富的告警与处理闭环。

## 八、REST 接口设计

以下为故障报修模块对外提供的 REST 接口，路径前缀为 `/api/fault`，无需版本前缀。

### 8.1 用户/前端报修

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/fault/report` | 提交故障工单（座位/区域/馆/设备至少填一类） |

**请求体示例**（JSON）：

```json
{
  "libraryId": 1,
  "areaId": null,
  "seatId": null,
  "equipmentId": null,
  "faultType": "seat_broken",
  "severity": "medium",
  "title": "3楼A区某座位椅面损坏",
  "description": "可选详细描述",
  "reportSource": "用户",
  "reportUserId": 10001
}
```

**约束**：`libraryId`、`areaId`、`seatId`、`equipmentId` 至少一个非空；`faultType` 取值：seat_broken / power_failure / env_issue / network_fault / other；`severity` 取值：low / medium / high；`reportSource` 取值：用户 / 监控 / 管理员 / 系统（或与后端约定的英文码一致）。创建后工单 **`status` 为 `REPORTED`**（界面展示「已报修」）。

**响应**：返回工单 id、status 等（与 8.2 详情结构一致）。

---

### 8.2 运维/管理员

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/fault/list` | 分页列表，支持按馆/区域/状态/类型/严重程度/时间范围筛选 |
| GET | `/api/fault/{id}` | 工单详情 |
| PATCH | `/api/fault/{id}/status` | 更新工单状态（`ACCEPTED` / `IN_PROGRESS` / `RESTORED` / `CLOSED`），可同时传 `resolvedTime`、`assignee`、**`adminRemark`**（写入 `admin_remark`） |

**GET /api/fault/list 查询参数建议**：

- `libraryId`、`areaId`、`status`、`faultType`、`severity`：精确筛选
- `startTime`、`endTime`：按创建时间范围
- `page`、`size`：分页，默认 page=1，size=10

**PATCH /api/fault/{id}/status 请求体示例**：

```json
{
  "status": "RESTORED",
  "resolvedTime": "2025-03-11T14:00:00",
  "assignee": "张三",
  "adminRemark": "已更换配件"
}
```

---

### 8.3 资源健康查询（供座位/调拨/UserDemand 等模块调用）

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/fault/health/query` | 批量查询资源是否因未关闭故障而不可用，用于排除坏座位、故障馆等 |

**请求体**（JSON）：

```json
{
  "resources": [
    { "resourceType": "LIBRARY", "resourceId": 1 },
    { "resourceType": "SEAT_AREA", "resourceId": 10 },
    { "resourceType": "SEAT", "resourceId": 1001 },
    { "resourceType": "DEVICE", "resourceId": 5 }
  ]
}
```

`resourceType` 取值：LIBRARY / SEAT_AREA / SEAT / DEVICE。

**响应体**（JSON）：

```json
{
  "results": [
    {
      "resourceType": "LIBRARY",
      "resourceId": 1,
      "available": true,
      "hasCriticalFault": false,
      "faultSummary": null
    },
    {
      "resourceType": "SEAT",
      "resourceId": 1001,
      "available": false,
      "hasCriticalFault": true,
      "faultSummary": "座位损坏，等待维修"
    }
  ]
}
```

是否参与“不可用”判断由配置控制（见 8.4）：仅当存在未关闭且严重程度在 `exclude-severities` 内的工单时，该资源 `available=false`，`hasCriticalFault` 与 `faultSummary` 据此填充。

---

### 8.4 业务规则配置（首版）

在 `application-business-rules.yml`（或等价配置文件）中增加故障相关规则，供 FaultService / 资源健康查询使用：

```yaml
library:
  business-rules:
    fault:
      # 参与“资源不可用”判断的工单状态（须与 tb_fault_report.status 英文码一致）
      active-statuses:
        - REPORTED
        - ACCEPTED
        - IN_PROGRESS
      # 参与“资源不可用”判断的严重程度（未关闭且在该列表内则排除）
      exclude-severities:
        - medium
        - high
      # 馆级：仅当存在“高”严重度未关闭工单时，该馆不参与调拨/推荐；为 false 时则中、高均排除
      library-exclude-only-high: true
```

**说明**：馆级与座位/区域/设备在 `library-exclude-only-high` 下的差异见 **3.6.1**。

## 九、故障报修智能体与其他智能体的联动

本节结合当前项目中的 **UserDemandAgent、InventoryAgent、座位模块（SeatAvailabilityService / SeatController）** 的实现方式，说明故障报修智能体与它们的**联动时机、调用关系与数据流**，便于在实现时在正确位置接入故障能力。

### 9.1 联动总览

| 模块 | 当前行为（简述） | 与故障报修的联动方式 |
|------|------------------|----------------------|
| **UserDemandAgent** | 轮询任务队列处理借书；本馆有书则直接借并返回 `recommendSeatLibraryId`；本馆无书则 Contract Net 选馆调拨，返回结果中同样带推荐取书馆。 | 在“确定推荐取书馆”后、写回结果前，对推荐馆做健康查询；若该馆不可用则可不推荐座位或改用备选馆。 |
| **InventoryAgent** | 在 DF 注册为 INVENTORY_SERVICE，响应 CFP 时用 `evaluateCapability(biblioId, userLibraryId)` 投标。 | 在 `evaluateCapability` 中，若本馆存在未关闭且符合配置的故障（馆级按 `library-exclude-only-high`），则拒绝投标或大幅降分，避免被选为调拨源馆。 |
| **座位模块** | `SeatAvailabilityService.queryAvailableSeats` 按馆查 OPEN 区域、AVAILABLE 座位，再按预约冲突过滤、排序、可选自动分配。 | 在得到“候选座位/区域”后，调用资源健康查询接口，排除有故障的座位/区域，再继续排序与分配。 |
| **FaultRepairAgent** | 接收报修、维护工单状态；对外提供健康查询与配置驱动的是否参与排除逻辑。 | 工单创建或状态变更为 `REPORTED` / `ACCEPTED` / `IN_PROGRESS` 且严重程度为中/高时，可向座位/调拨侧发送资源影响通知，便于其刷新缓存或重算。 |

以下按模块分别说明**具体集成点**与**推荐实现方式**（以“先 REST/本地服务，再扩展 JADE 消息”的渐进方式为准）。

---

### 9.2 与 UserDemandAgent 的联动

**当前代码要点**（`UserDemandAgent`）：

- 借书任务来自 `AgentTaskManager.pollTask()`，参数包含 `userId, biblioId, preferredLibraryId`。
- **场景 A**：本馆有书 → 直接借书成功 → 结果中写入 `recommendSeatLibraryId = userLibraryId`，并带 `recommendSeatDate`、`actionRecommendSeat` 等，表示推荐用户在该馆预约座位。
- **场景 B**：本馆无书 → Contract Net 与各馆 `InventoryAgent` 协商 → 选评分最高的馆调拨 → 结果中不直接写 `recommendSeatLibraryId`（取书馆一般为 `userLibraryId`，即用户期望取书馆）；若业务上也会在调拨结果里推荐“到馆取书后预约座位”，则推荐馆同样是取书馆（用户所在馆）。

**联动目的**：避免把用户推荐到“因故障暂时不宜使用”的馆（例如全馆或主区维修、网络瘫痪等），从而与“书籍跨馆调配 – 座位动态分配”一致地考虑馆级健康。

**集成点与建议实现**：

1. **在写回“借书/调拨成功”结果、且结果中包含 `recommendSeatLibraryId` 之前**，对该馆做一次馆级健康查询。
   - 调用方式：在 UserDemandAgent 内通过 Spring 注入或 `SpringContextUtil.getBean()` 获取**资源健康查询服务**（封装 `POST /api/fault/health/query` 或直接调用 `FaultService` 的批量健康查询方法）。
   - 请求：`resources = [{ resourceType: "LIBRARY", resourceId: recommendSeatLibraryId }]`。
   - 若响应中该馆 `available === false`（即存在未关闭且符合配置的馆级故障）：
     - **方案 1**：将 `recommendSeatLibraryId` 置空或不带座位推荐字段，并在 `message` 中提示“当前馆区维护中，暂不推荐座位，请稍后再试或到馆后现场选座”。
     - **方案 2**：若有“备选馆”列表，可对备选馆再查健康，选第一个 `available === true` 的馆作为 `recommendSeatLibraryId`（仅当业务允许推荐到非取书馆时使用）。
2. 若项目后续引入 **FaultRepairAgent 的 JADE 通信**，UserDemandAgent 可在同一决策点改为发送 **RESOURCE_HEALTH_QUERY** 给 FaultRepairAgent，根据 **RESOURCE_HEALTH_RESPONSE** 再决定是否推荐该馆；业务逻辑与上述一致。

这样，故障报修智能体与 UserDemandAgent 的联动**仅发生在“是否推荐某馆作为座位推荐馆”**，不改变借书/调拨主流程，也不替代 InventoryAgent 的投标逻辑。

---

### 9.3 与 InventoryAgent（调拨决策）的联动

**当前代码要点**（`InventoryAgent`）：

- 每馆一个实例，在 `handleCFP` 中根据 `evaluateCapability(biblioId, userLibraryId)` 决定是否投标（PROPOSE）及评分。
- 评分考虑：本馆是否有库存、是否用户所在馆（本地借阅满分）、否则为调拨方案（距离、压力等）。

**联动目的**：当**本馆**存在未关闭的馆级故障（如全馆维修、重大设备故障）时，该馆不应作为调拨的**源馆**或**目标馆**，避免书调到“暂时无法正常服务”的馆。

**集成点与建议实现**：

1. **在 `evaluateCapability` 开头或返回 `canServe = true` 之前**，增加“本馆健康”检查：
   - 调用资源健康查询服务，查询 `resources = [{ resourceType: "LIBRARY", resourceId: libraryId }]`（本馆 ID）。
   - 配置含义：若 `library-exclude-only-high: true`，仅当存在**高**严重度未关闭工单时，该馆视为不可用；若为 `false`，则中、高严重度均视为不可用。
   - 若该馆 `available === false`：
     - **推荐**：直接设置 `canServe = false`，`reason = "本馆当前处于维护/故障状态，暂不参与调拨"`，不发送 PROPOSE，与“无库存”一样被 UserDemandAgent 视为无法服务。
     - **可选**：若希望“降权而非完全排除”，可保留 `canServe = true` 但大幅降低 `score`（例如减 50 分），使该馆仅在无其他馆可选时才被选中。
2. 若后续采用 JADE 协议，InventoryAgent 可在评估前向 FaultRepairAgent 发送 **RESOURCE_HEALTH_QUERY**（馆维度），根据 **RESOURCE_HEALTH_RESPONSE** 再决定是否投标及评分。

这样，**调拨决策侧**在“选馆”时就已经把故障状态考虑进去，与故障报修智能体的数据源（tb_fault_report + 配置）保持一致。

---

### 9.4 与座位模块（SeatAvailabilityService）的联动

**当前代码要点**（`SeatAvailabilityService.queryAvailableSeats`）：

- 入参：`libraryId, reservationDate, startTime, endTime, areaId, hasPower, autoAssign`。
- 流程：按馆查 `OPEN` 的 `SeatArea` → 再查这些区域下 `status = 'AVAILABLE'` 的 `Seat` → 按当日 ACTIVE 预约做时间冲突过滤 → 排序（有电源优先、楼层等）→ 若 `autoAssign` 则返回第一个座位，否则返回列表。

**联动目的**：有未关闭且中/高严重度故障的**座位**应从候选中剔除；若**区域**存在此类故障，可整区不参与推荐或降权，避免用户被分配到“坏座位”或“施工中区域”。

**集成点与建议实现**：

1. **在得到“候选座位列表”且做完“预约冲突”过滤之后、排序与分配之前**，增加故障过滤：
   - 收集当前候选的 `seatIds` 以及它们所属的 `areaIds`（可从现有 `areaMap`、`seatList` 得到）。
   - 调用资源健康查询：  
     `resources = seatIds.map(id => ({ resourceType: "SEAT", resourceId: id }))`  
     并加上 `areaIds` 去重：`areaIds.map(id => ({ resourceType: "SEAT_AREA", resourceId: id }))`。  
     若希望馆级故障也影响“整馆不开放座位”，可再追加 `{ resourceType: "LIBRARY", resourceId: libraryId }`。
   - 根据响应：
     - **座位维度**：若某 `seatId` 的 `available === false`，从候选列表移除该座位。
     - **区域维度**：若某 `areaId` 的 `available === false`，可将该区域下所有候选座位移除（或仅移除该区，视产品需求而定）。
     - **馆维度**：若 `libraryId` 的 `available === false`，可直接返回空结果或提示“该馆当前维护中”。
   - 对过滤后的候选列表继续执行现有排序与 `autoAssign` 逻辑。
2. 实现方式：在 `SeatAvailabilityService` 中注入 **FaultService 或专门的 ResourceHealthQueryService**，封装对 `POST /api/fault/health/query` 的调用（或内网直接查 FaultService），避免 Controller 层关心故障逻辑。

这样，**座位可用性与动态分配**在现有“区域开放 + 座位状态 + 预约冲突”之外，增加“故障排除”一层，与故障报修智能体的工单数据与配置一致。

### 9.5 FaultRepairAgent 的主动通知（可选）

**当前设计**：故障报修以“工单 + 健康查询接口”为主，其他模块在**自身决策时主动查询**即可完成联动。

若希望**故障状态变更时**座位/调拨侧能更快刷新（例如不依赖下次请求再查），可增加“主动通知”：

1. **触发时机**：工单新建或状态变更为 `REPORTED` / `ACCEPTED` / `IN_PROGRESS`，且严重程度为配置中的 `exclude-severities`（如 medium/high）；或工单状态变更为 `RESTORED` / `CLOSED`。
2. **通知内容**：与第五章协议一致，例如 **RESOURCE_IMPACT_NOTIFICATION**，包含 `resourceType、resourceId、event（如 SEAT_UNAVAILABLE / SEAT_AVAILABLE）、faultId、newStatus` 等。
3. **接收方与行为**：
   - **座位模块**：若存在“可用座位/区域缓存”，可监听该通知并失效对应缓存，或标记该座位/区域为待刷新；下次 `queryAvailableSeats` 会自然通过健康查询拿到最新结果。若当前项目暂无缓存，仅依赖每次请求时查健康，则首版可不实现监听，仅保留“按请求查询”即可。
   - **UserDemandAgent / InventoryAgent**：通常不做长期缓存，每次借书/投标时现场查馆健康即可；若后续为性能引入馆级健康缓存，可同样监听馆维度的 RESOURCE_IMPACT_NOTIFICATION 做失效。

实现上，FaultRepairAgent 在工单创建/更新后，可调用 Spring 事件发布（`ApplicationEventPublisher.publishEvent(ResourceFaultChangedEvent)`），由座位或调拨侧的监听器统一处理；或通过 JADE ACL 向已注册的“座位代理”“调拨协调者”发送消息，便于与现有 Agent 架构对齐。

### 9.6 联动小结（按实现顺序）

| 优先级 | 联动对象 | 动作 | 说明 |
|--------|----------|------|------|
| 1 | SeatAvailabilityService | 在候选座位/区域确定后调用健康查询，排除故障座位与故障区域 | 直接减少“坏座位/施工区”被推荐，与现有流程兼容 |
| 2 | InventoryAgent | 在 evaluateCapability 中查本馆健康，不可用时拒绝投标或大幅降分 | 避免调拨选到故障馆 |
| 3 | UserDemandAgent | 在返回带 recommendSeatLibraryId 的结果前查该馆健康，不可用时不再推荐座位或提示 | 保证推荐取书馆与座位推荐一致考虑故障 |
| 4 | FaultRepairAgent | 工单状态变更时发送 RESOURCE_IMPACT_NOTIFICATION 或 Spring 事件 | 可选，用于缓存失效或实时感知，首版可省略 |

以上联动均依赖**同一套资源健康查询语义**（tb_fault_report + 配置中的 active-statuses、exclude-severities、library-exclude-only-high），便于在实现与测试时保持行为一致，并便于后续扩展 JADE 消息协议而不改业务含义。
