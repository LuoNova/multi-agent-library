# 批量调拨功能修复文档

## 一、问题描述

### 1.1 问题现象
调拨建议建议调拨5本图书,审批通过后只调拨了1本:
- 被调拨馆库存只减少了1本
- `tb_book_transfer`表中只有1个副本状态变为`IN_TRANSIT`

### 1.2 问题根因
**数据库设计缺陷:**
- `tb_transfer_suggestion`表的`transfer_id`字段只能关联一个调拨记录
- `executeSuggestion`方法虽然循环创建了多个调拨记录,但`transfer_id`被反复覆盖
- 最终只保存了最后一个调拨记录ID,导致只执行了最后一个

**原有数据流程:**
```
TransferSuggestion (建议调拨5本)
    ↓ transfer_id (只能存一个ID)
BookTransfer (只关联了最后一个调拨记录)
```

## 二、解决方案

### 2.1 设计思路
引入**调拨单表**(`tb_transfer_order`)作为调拨建议和具体调拨记录之间的桥梁:

```
TransferSuggestion (调拨建议,建议调拨5本)
    ↓ 审批通过
TransferOrder (调拨单,记录批量调拨的整体信息)
    ↓ 执行
BookTransfer (调拨记录,每本副本一条记录,共5条,都关联同一个order_id)
```

### 2.2 核心改进
1. **批量调拨管理**: 通过`TransferOrder`统一管理一次批量调拨
2. **数据一致性**: 所有调拨记录共享同一个`order_id`,确保关联关系正确
3. **统计准确**: 调拨单记录计划数量和实际数量,便于审计和统计
4. **向后兼容**: 保留原有方法签名,不影响现有功能

2. **重写executeSuggestion方法:**

**核心逻辑:**
```java
//执行调拨建议（自动审批模式）
@Override
@Transactional
public void executeSuggestion(Long suggestionId) {
    TransferSuggestion suggestion = suggestionMapper.selectById(suggestionId);
    if (suggestion == null) {
        log.error("调拨建议不存在: {}", suggestionId);
        return;
    }

    //允许PENDING或APPROVED状态执行
    if (!"PENDING".equals(suggestion.getStatus()) && !"APPROVED".equals(suggestion.getStatus())) {
        log.error("调拨建议状态不正确: {}", suggestion.getStatus());
        return;
    }

    log.info("开始执行调拨建议: {}, 计划调拨{}本", suggestionId, suggestion.getSuggestedQuantity());

    //1.查找源馆的可用副本
    List<BookCopy> availableCopies = copyMapper.selectList(
            new LambdaQueryWrapper<BookCopy>()
                    .eq(BookCopy::getBiblioId, suggestion.getBiblioId())
                    .eq(BookCopy::getLibraryId, suggestion.getFromLibraryId())
                    .eq(BookCopy::getStatus, LibraryConstants.COPY_STATUS_AVAILABLE)
                    .last("LIMIT " + suggestion.getSuggestedQuantity())
    );

    if (availableCopies.isEmpty()) {
        log.error("源馆没有可用的副本");
        suggestion.setStatus("REJECTED");
        suggestion.setReason("源馆没有可用的副本");
        suggestionMapper.updateById(suggestion);
        return;
    }

    //2.创建调拨单（批量调拨的主记录）
    TransferOrder order = new TransferOrder();
    order.setSuggestionId(suggestionId);
    order.setFromLibraryId(suggestion.getFromLibraryId());
    order.setToLibraryId(suggestion.getToLibraryId());
    order.setBiblioId(suggestion.getBiblioId());
    order.setPlannedQuantity(suggestion.getSuggestedQuantity());
    order.setActualQuantity(availableCopies.size()); //实际可调拨数量
    order.setStatus("IN_PROGRESS"); //执行中
    order.setCreateTime(LocalDateTime.now());
    orderMapper.insert(order);
    log.info("创建调拨单成功: orderId={}, 计划{}本, 实际{}本",
            order.getId(), order.getPlannedQuantity(), order.getActualQuantity());

    //3.为每个副本创建调拨记录并执行
    int successCount = 0;
    for (BookCopy copy : availableCopies) {
        //创建调拨记录（使用增强方法，指定调拨原因、建议ID和调拨单ID）
        Long transferId = transferService.createTransfer(
                copy.getId(),
                suggestion.getFromLibraryId(),
                suggestion.getToLibraryId(),
                null, //没有预约ID
                "INVENTORY_BALANCE", //调拨原因：库存平衡
                suggestionId, //关联的调拨建议ID
                order.getId() //关联的调拨单ID
        );

        log.info("调拨记录创建成功：transferId={}, copyId={}", transferId, copy.getId());

        //执行调拨：更新副本状态为IN_TRANSIT，源馆库存-1
        boolean success = bookCopyService.executeTransfer(
                copy.getId(),
                suggestion.getFromLibraryId(),
                suggestion.getToLibraryId(),
                suggestion.getBiblioId()
        );

        if (!success) {
            log.error("副本{}调拨执行失败，副本状态和库存未更新", copy.getId());
            //继续处理其他副本，不中断整个流程
        } else {
            successCount++;
            log.info("副本{}调拨执行成功，状态已更新为IN_TRANSIT，源馆库存已减少", copy.getId());
        }
    }

    //4.更新调拨单状态
    order.setStatus("COMPLETED");
    order.setActualQuantity(successCount);
    order.setCompleteTime(LocalDateTime.now());
    orderMapper.updateById(order);
    log.info("调拨单执行完成: orderId={}, 成功调拨{}本", order.getId(), successCount);

    //5.更新建议状态
    suggestion.setStatus("EXECUTED");
    suggestion.setOrderId(order.getId()); //关联调拨单
    suggestion.setUpdateTime(LocalDateTime.now());
    suggestionMapper.updateById(suggestion);

    log.info("调拨建议执行完成: suggestionId={}, orderId={}", suggestionId, order.getId());
}
```

#### 3.2.4 TransferService.java

```java
//创建调拨记录（完整方法，支持指定调拨原因、建议ID和调拨单ID）
public Long createTransfer(Long copyId, Long fromLibraryId, Long toLibraryId, Long reservationId,
                           String transferReason, Long suggestionId, Long orderId) {
    log.info("创建调拨记录：副本{}从馆{}到馆{}，关联预约{}，调拨原因{}，建议ID{}，调拨单ID{}",
            copyId, fromLibraryId, toLibraryId, reservationId, transferReason, suggestionId, orderId);

    BookTransfer transfer = new BookTransfer();
    transfer.setCopyId(copyId);
    transfer.setFromLibraryId(fromLibraryId);
    transfer.setToLibraryId(toLibraryId);
    transfer.setStatus(LibraryConstants.TRANSFER_STATUS_IN_TRANSIT);
    transfer.setRequestTime(LocalDateTime.now());
    //预计30分钟后到达（用于定时任务自动完成）
    transfer.setCompleteTime(LocalDateTime.now().plusMinutes(
            businessRulesProperties.getTransfer().getEstimatedMinutes()));
    //设置调拨原因、建议ID和调拨单ID
    transfer.setTransferReason(transferReason);
    transfer.setSuggestionId(suggestionId);
    transfer.setOrderId(orderId);

    transferMapper.insert(transfer);
    log.info("调拨记录创建成功：{}", transfer.getId());

    return transfer.getId();
}
```

## 四、数据库表结构

### 4.1 表关系图
```
tb_transfer_suggestion (调拨建议)
    ↓ order_id (外键)
tb_transfer_order (调拨单)
    ↓ id (主键)
tb_book_transfer (调拨记录,多条记录共享同一个order_id)
```

### 4.2 调拨单表 (tb_transfer_order)

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | bigint | 主键ID |

- 5个副本状态都变为IN_TRANSIT
- 调拨单表记录批量调拨信息
- 调拨记录表有5条记录,都关联同一个调拨单

### 6.2 数据一致性

**调拨单表:**
- 记录批量调拨的整体信息
- 计划数量 = 5
- 实际数量 = 5
- 状态 = COMPLETED

**调拨记录表:**
- 5条记录,每条记录对应一个副本
- 所有记录的`order_id`相同
- 所有记录的`status` = IN_TRANSIT
- 所有记录的`transfer_reason` = INVENTORY_BALANCE

**副本表:**
- 5个副本状态变为IN_TRANSIT
- 副本仍在源馆,等待运输

**库存统计表:**
- 源馆库存减少5本
- 目标馆库存不变(等待调拨完成)

### 6.3 可追溯性

**完整的数据链路:**
```
调拨建议(id=1) 
  → 调拨单(id=1, planned_quantity=5, actual_quantity=5)
    → 调拨记录(id=1-5, order_id=1)
      → 副本(copy_id=101-105, status=IN_TRANSIT)
        → 库存变化(馆2: 10→5)
```

**查询示例:**
```sql
-- 查询某个调拨建议的所有调拨记录
SELECT bt.* 
FROM tb_book_transfer bt
JOIN tb_transfer_order tord ON bt.order_id = tord.id
WHERE tord.suggestion_id = 1;

-- 查询某个调拨单的执行情况
SELECT 
  tord.id AS order_id,
  tord.planned_quantity,
  tord.actual_quantity,
  COUNT(bt.id) AS transfer_count
FROM tb_transfer_order tord
LEFT JOIN tb_book_transfer bt ON tord.id = bt.order_id
WHERE tord.id = 1
GROUP BY tord.id;
```

### 7.1 数据库迁移

**如果表已存在:**
- 需要先备份数据
- 使用ALTER语句修改表结构,而不是DROP后重建
- 注意数据迁移,特别是`transfer_id` → `order_id`的转换

**ALTER语句示例:**
```sql
-- 修改调拨建议表
ALTER TABLE `tb_transfer_suggestion`
DROP COLUMN `transfer_id`,
ADD COLUMN `order_id` bigint DEFAULT NULL COMMENT '关联的调拨单ID' AFTER `approve_time`;

-- 修改调拨记录表
ALTER TABLE `tb_book_transfer`
ADD COLUMN `order_id` bigint DEFAULT NULL COMMENT '关联的调拨单ID' AFTER `suggestion_id`;

-- 添加索引
ALTER TABLE `tb_book_transfer`
ADD INDEX `idx_order` (`order_id`);
```
