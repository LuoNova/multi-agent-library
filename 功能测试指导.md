# 功能测试指导文档

本文档详细说明如何测试多智能体图书馆资源调度系统的各项功能。

## 一、测试环境准备

### 1.1 数据库准备

在开始测试前，请先执行测试数据脚本：

```bash
# 在MySQL中执行测试数据脚本
mysql -u root -p library_db < test_data.sql
```

或者使用数据库管理工具（如Navicat、MySQL Workbench）执行`test_data.sql`文件。

### 1.2 启动应用

```bash
# 进入项目目录
cd D:/javaProjects/multi-agent-library

# 启动Spring Boot应用
mvn spring-boot:run
```

等待应用启动完成，看到以下日志表示启动成功：
```
Started MultiAgentLibraryApplication in X.XXX seconds
JADE Agents started successfully
```

### 1.3 访问Swagger文档

启动成功后，访问Swagger UI查看和测试API接口：
```
http://localhost:8080/swagger-ui.html
```

## 二、功能测试场景

### 功能1：智能借书与资源调度

#### 测试场景1.1：本地借阅（目标馆有库存）

**测试目的**：验证当目标馆有库存时，系统能够直接办理借阅。

**前置条件**：
- 用户ID=1（张三），状态正常，借阅计数为0
- 书目ID=1（《Java编程思想》）
- 理科馆（ID=1）有该书的可用副本

**测试步骤**：

1. 打开Swagger UI，找到`POST /api/borrow/request`接口
2. 输入以下请求参数：
```json
{
  "userId": 1,
  "biblioId": 1,
  "preferredLibraryId": 1
}
```
3. 点击"Execute"执行请求

**预期结果**：
- HTTP状态码：200
- 响应消息：`"借书成功，本地借阅"`
- 响应数据包含：
  - `status`: "SUCCESS"
  - `strategy`: "LOCAL_LOAN"
  - `borrowId`: 借阅记录ID（数字）
  - `copyId`: 副本ID（数字）

**验证方法**：
```sql
-- 查询借阅记录
SELECT * FROM tb_book_borrow WHERE user_id = 1 ORDER BY create_time DESC LIMIT 1;

-- 查询副本状态（应该变为BORROWED）
SELECT * FROM tb_book_copy WHERE id = <返回的copyId>;

-- 查询用户借阅计数（应该+1）
SELECT current_borrow_count FROM tb_user WHERE id = 1;
```

### 功能5：预约过期自动释放

**测试目的**：验证定时任务自动释放超期未取的预留记录。

**前置条件**：
- 借阅记录状态为RESERVED
- 取书截止时间已过


## 六、常见问题排查

### 6.1 借书请求超时

**可能原因**：
- 多智能体协商时间过长
- 数据库连接超时
- JADE Agent未正常启动

**排查方法**：
1. 检查JADE Agent是否正常启动（查看日志）
2. 检查数据库连接是否正常
3. 查看BorrowController中的超时设置（默认5秒）

### 6.2 调拨单无法完成

**可能原因**：
- 调拨单状态不是IN_TRANSIT
- 关联的借阅记录或预约记录不存在

**排查方法**：
```sql
-- 查询调拨单状态
SELECT * FROM tb_book_transfer WHERE id = <transferId>;

-- 查询关联的借阅记录
SELECT * FROM tb_book_borrow WHERE copy_id = <copyId> AND status = 'TRANSFERRING';

-- 查询关联的预约记录
SELECT * FROM tb_book_reservation WHERE copy_id = <copyId> AND status = 'FULFILLED';
```

### 6.3 取书确认失败

**可能原因**：
- 借阅记录状态不是RESERVED
- 已超过取书截止时间
- 用户ID不匹配

**排查方法**：
```sql
-- 查询借阅记录详情
SELECT * FROM tb_book_borrow WHERE id = <borrowId>;

-- 检查取书截止时间
SELECT pickup_deadline FROM tb_book_borrow WHERE id = <borrowId>;
```

### 功能5：主动调拨与库存平衡

#### 测试场景5.1：手动触发库存平衡分析

**测试目的**：验证系统能够正确分析库存并生成调拨建议。

**前置条件**：
- 数据库中存在库存不均衡的情况
- 例如：理科馆某书库存不足，文科馆该库存充足

**准备数据**：
```sql
-- 设置理科馆《Java编程思想》库存不足且需求旺盛
UPDATE tb_library_biblio_stats 
SET stock_count = 1, borrow_count_30d = 25, reservation_pending_count = 5
WHERE library_id = 1 AND biblio_id = 1;

-- 设置文科馆《Java编程思想》库存充足且利用率低
UPDATE tb_library_biblio_stats
```

   - `GET /api/seat/available`  
   - 参数示例：
     - `libraryId = 1`  
     - `reservationDate = 明天`  
     - `startTime = 09:00`  
     - `endTime = 11:00`

**预期结果：**

- `code = 200`；  
- 若 `autoAssign=true`（默认），`data.availableTotal = 0`，`data.assignedSeat = null`，`message = "当前条件下无可用座位"`；  
- 若显式 `autoAssign=false`，`data.total = 0`，`data.seats` 为空数组，`message = "当前条件下无可用座位"`。

#### 场景5：参数非法校验

**目的**：验证最小/最大时长和步长配置生效。

**示例1：时长过短**

- 参数：
  - `startTime = 09:00`  
  - `endTime = 09:30`（小于 `min-duration-minutes=60`）

**预期结果：**

- `code = 400`；  
- `message` 包含 `"预约时长必须在"` 的提示。

**示例2：非步长整数倍**

- 参数：
  - `startTime = 09:00`  
  - `endTime = 10:30`（时长 90 分钟，不是 `slot-step-minutes=60` 的整数倍）

**预期结果：**

- `code = 400`；  
- `message` 包含 `"预约时长必须是60分钟的整数倍"`。

**示例3：日期超出可预约范围**

- 参数：
  - `reservationDate = 今天 - 1 天` 或 `今天 + 3 天`（超出 `0~2` 天范围）

**预期结果：**

- `code = 400`；  
- `message` 包含 `"预约日期不在允许范围内"`。

### 测试总结

通过以上场景，基本可以覆盖座位可用性查询与动态分配的核心逻辑，包括：

- 配置驱动的时间范围与时长校验；
- 按馆+区域过滤、避免跨馆选座；
- 按预约记录过滤时间冲突；
- 默认自动分配与列表模式的行为差异；
- 无可用座位时的返回约定。

建议在本地验证通过后，再在集成环境结合前端页面进行一次端到端联调测试。

## 十二、创建座位预约测试（功能2）

### 测试目标

验证 `POST /api/seat/reservation` 能正确创建座位预约、占用座位，并覆盖：参数与时间校验、馆开放时间、座位/区域状态、用户维度冲突、座位维度冲突、source 推断及返回数据。

### 测试前准备

1. 确保已执行座位测试数据脚本，存在 `library_id=1` 下 `status='AVAILABLE'` 的座位（如 `seatId=1002`）。
2. 馆开放时间：若需校验开放时间，请保证 `tb_library` 中对应馆的 `open_time` 为 `08:00-22:00` 等格式；为空则不校验时段。
3. Swagger：`http://localhost:8080/swagger-ui.html` → 座位管理 → 创建座位预约。

### 测试场景

#### 场景1：正常创建预约

**步骤：**

1. 调用 `GET /api/seat/available`，确认某座位可用（如 `libraryId=1`，明天，`09:00`-`11:00`）。
2. 调用 `POST /api/seat/reservation`，请求体示例：

```json
{
  "userId": 1,
  "seatId": 1002,
  "libraryId": 1,
  "reservationDate": "2026-03-11",
  "startTime": "09:00",
  "endTime": "11:00"
}
```

（将 `reservationDate` 改为明天日期）

**预期结果：**

- `code = 200`，`message = "座位预约创建成功"`。
- `data` 包含 `reservationId`、`seatId`、`areaId`、`areaName`、`floor`、`seatNo`、`hasPower`、`reservationDate`、`startTime`、`endTime`、`source="WALK_IN"`。
- 数据库中 `tb_seat_reservation` 新增一条 `status='ACTIVE'` 记录，`library_id`、`source` 正确。
- 对应 `tb_seat` 的 `status` 变为 `OCCUPIED`。

#### 场景2：带 borrowId 时 source 推断为 BORROW_PICKUP

**步骤：** 同上，请求体增加 `"borrowId": 1`，不传 `source`。

**预期结果：** `data.source = "BORROW_PICKUP"`，表中 `borrow_id=1`，`source='BORROW_PICKUP'`。

#### 场景3：用户维度冲突（同一用户同馆同日同时段已有一条预约）

**步骤：**

1. 先为用户1创建一条预约（如座位 1003，明天 14:00-16:00）。
2. 再为用户1创建另一条预约：同一馆、同一天、时间段重叠（如明天 15:00-17:00 或 14:30-15:30）。

**预期结果：**

- `code = 400`，`message = "您在该时间段已预约其他座位,请先取消或调整时间"`。
- 未新增第二条预约，座位状态不变。

#### 场景4：座位维度冲突（该座位该时段已被他人预约）

**步骤：**

1. 用户1已预约座位 1004 明天 10:00-12:00。
2. 用户2请求预约同一座位、同一天、同一时段（或重叠时段）。

**预期结果：**

- `code = 400`，`message = "该时间段座位已被占用,请更换时间或座位"`。
- 未新增预约。

#### 场景5：座位不可用或不属于该馆

**步骤：** 将已占用的座位（`status='OCCUPIED'`）或非本馆座位（如 `seatId` 属于馆2，`libraryId=1`）用于创建预约。

**预期结果：** `code = 400`，提示“座位不存在或不可用”或“座位不属于该馆”。

#### 场景6：参数非法（时长、日期、开放时间）

- 时长过短：`startTime="09:00"`, `endTime="09:30"` → 400，提示预约时长必须在 60~840 分钟之间。
- 非步长整数倍：`endTime="10:30"`（90 分钟）→ 400，提示 60 分钟整数倍。
- 日期超出 0~2 天：`reservationDate` 为昨天或 3 天后 → 400，预约日期不在允许范围内。
- 馆开放时间：若馆 `open_time="08:00-22:00"`，请求 `startTime="07:00"` 或 `endTime="23:00"` → 400，提示预约时段不在馆开放时间内。
