# 前端「查询回显」与后端接口对照

> 目的：梳理当前仓库 **Spring Boot Controller** 已暴露的**只读/列表类**接口，与 **Vue 前端各功能**做对照，便于实现「先查询 → 表格/卡片回显 → 再调原有写接口」的交互。  
> 依据：`src/main/java/com/library/controller/*.java`、`frontend/src/api/*.ts`、根目录 `API接口文档-前后端对接.md`。  
> **2026-05 更新**：已新增 **`/api/catalog/*`** 只读接口（见根目录 **`功能文档.md`**「功能9」），下表读者端借/还/取书、座位选馆/区/座、设备列表等已从「尚无」变为「已有」。  
> **未包含**：`/test/*` 测试接口（非正式业务 API）；前端当前未接入的 `/api/notification/template/*`。

---

## 1. 图例

| 标记 | 含义 |
|------|------|
| **已有** | 后端已有可用的 GET（或 POST 但语义为「查询」如健康查询），前端可直接对接做列表/详情回显。 |
| **部分已有** | 有一部分查询能力，但关键维度仍缺接口，或现有接口存在实现缺口。 |
| **尚无** | 当前 Controller 中**无**对应只读接口，要做真实「查询回显」需后端新增（或改现有接口返回结构）。 |

| 管理端 | 调拨：到货/批量 | `POST /api/transfer/complete*` | 调拨单/记录列表 | **已有**（`GET /api/catalog/transfers` 富展示分页 + 行内填入；亦可 `GET /api/transfer/list`） |
| 管理端 | 调拨：进度 | `GET /api/transfer/progress/*` | 记录 ID | **已有**（可与列表联动） |
| 管理端 | 调拨：建议 | `GET/POST .../suggestion/*` | 建议列表、详情 | **已有**（`GET .../list`；`GET .../{id}` 已按主键查询补全） |
| 管理端 | 故障运维 | `GET/PATCH/POST health` | 工单列表、详情 | **已有**（`GET /api/catalog/fault-tickets` 含馆名 + 馆下拉；详情/更新仍 `GET/PATCH /api/fault/*`；健康 `POST /api/fault/health/query`） |
| 管理端 | 设备状态 | `PATCH /api/equipment/{id}/status` | 设备列表或搜索 | **已有**（`GET /api/catalog/equipment`） |
| 管理端 | 定时任务调试 | `POST /api/task/trigger/*` | 一般无需列表 | **不适用**（无「查询回显」强需求） |

## 3. 读者端（`/u/*`）明细

### 3.1 借书申请（`BorrowView` → `POST /api/borrow/request`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 选书（替代 `biblioId`） | **`GET /api/catalog/biblios`** |
| 选取书馆（替代 `preferredLibraryId`） | **`GET /api/catalog/libraries`** |

**结论**：已实现前后端联动；条码单独解析副本未做（可按需再加 `GET /api/catalog/copy-by-barcode` 等）。

---

### 3.2 还书（`ReturnView` → `POST /api/book/return`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 在借行（含 `copyId`） | **`GET /api/catalog/borrows/active?userId=`** |
| 还书馆 | **`GET /api/catalog/libraries`**（列表行中带默认还书馆 ID/名称） |

**结论**：已实现「在借表格 + 详情 + 还书弹窗选馆」。

---

### 3.3 取书确认（`PickupView` → `POST /api/borrow/confirm-pickup`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 待取书列表 | **`GET /api/catalog/borrows/pickup-pending?userId=`**（`RESERVED` / `TRANSFERRING`） |

**结论**：已实现表格选单；仅 `RESERVED` 可点「确认取书」（与后端校验一致）。

---

### 3.3.1 我的借阅记录（`BorrowHistoryView` → 只读）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 历史借阅流水（多状态）、分页与状态筛选 | **`GET /api/catalog/borrows/history?userId=&status=&page=&size=`** |

**结论**：读者端菜单「我的借阅记录」；与「故障工单处理记录」不是同一概念，后者需故障模块扩展。

---

### 3.4 我的调拨与进度（`TransferUserView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 我的调拨列表 | `GET /api/transfer/my-transfers` | **已有**，可做表格，`records` 内字段可用于再查进度 |
| 单条进度 | `GET /api/transfer/progress/{transferId}` | **已有** |
| 批量单进度 | `GET /api/transfer/progress/batch/{orderId}` | **已有** |

**结论**：查询回显 **已具备**；前端可把「手输 transferId」改为从 `my-transfers` 的 `records` 中选（或详情抽屉再调进度 GET）。

---

### 3.5 座位（`SeatView`）

| Tab / 能力 | 方法 | 说明 |
|--------------|------|------|
| 查询可用 | `GET /api/seat/available` | **已有**，返回座位列表或自动分配结果，可直接驱动「选座」 |
| 我的预约 | `GET /api/seat/reservation/my` | **已有**，可表格 + 行内取消/签到等 |
| 取消/签到/暂离/结束 | 多个 `POST .../reservation/*` | 写接口已有；`reservationId` 可从「我的预约」行带出 |

**结论**：馆/区/座位已用 **`/api/catalog/libraries`、`/seat-areas`、`/seats`**；查可用仍用 **`/api/seat/available`**。

---

### 3.6 故障报修（`FaultReportView` → `POST /api/fault/report`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 馆 / 区 / 座 | **`/api/catalog/libraries`、`/seat-areas`、`/seats`** |
| 设备 | **`/api/catalog/equipment`** 分页点选 |

说明：`GET /api/fault/list` 仍为工单列表，与物理资源选择无关。

**结论**：已实现级联与设备表选用。

---

## 4. 管理端（`/a/*`）明细

### 4.1 调拨与库存（`TransferAdminView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 全量调拨记录（表格回显） | **`GET /api/catalog/transfers`** | **已有**，`records` 含馆名、书名、`orderId`；页面内可填入到货/进度/批量参数 |
| 全量调拨记录（原始实体） | `GET /api/transfer/list` | **已有**，与上者筛选一致，字段为实体 |
| 用户调拨（若需对照） | `GET /api/transfer/my-transfers` | **已有**（需 `userId` Query，偏运维查询） |
| 单条/批量进度 | `GET /api/transfer/progress/{transferId}`、`GET .../batch/{orderId}` | **已有** |
| 调拨建议列表 | `GET /api/transfer/suggestion/list` | **已有** |
| 调拨建议详情 | `GET /api/transfer/suggestion/{id}` | **已有**（按主键查询；不存在时 `code=404`） |

写操作：`POST /api/transfer/complete`、`POST /api/transfer/complete-batch`、建议审批/拒绝/触发平衡等 **已有**。

---

### 4.2 故障工单运维（`FaultManageView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 工单分页（表格 + 馆名） | **`GET /api/catalog/fault-tickets`** | **已有**，筛选与 `GET /api/fault/list` 一致，`records` 含 `libraryName`；列表页配合 **`GET /api/catalog/libraries`** 下拉 |
| 工单分页（原始 VO） | `GET /api/fault/list` | **已有** |
| 工单详情 | `GET /api/fault/{id}` | **已有** |
| 资源健康 | `POST /api/fault/health/query` | **已有**（请求体带资源列表，属「查询类」） |

**结论**：列表以 **`/api/catalog/fault-tickets`** 为主做回显；写状态仍走 **`PATCH /api/fault/{id}/status`**。

---

### 4.3 设备状态（`EquipmentView` → `PATCH /api/equipment/{id}/status`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 选设备再改状态 | **`GET /api/catalog/equipment`** | 分页 + 可选 `libraryId` 筛选 |

**结论**：已实现列表点选填入设备 ID 再 PATCH。

---

### 4.4 定时任务调试（`DevTasksView`）

仅 `POST /api/task/trigger/*`，无列表依赖。

**结论**：**不适用**「查询回显」；最多展示上次返回文案（前端状态即可）。

## 5. 后端已有 GET / 查询类接口速查（便于排期）

| GET | `/api/fault/{id}` | 工单详情 |
| POST | `/api/fault/health/query` | 资源健康（查询语义） |
