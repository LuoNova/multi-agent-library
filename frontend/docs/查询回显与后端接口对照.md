# 前端「查询回显」与后端接口对照

> 目的：梳理当前仓库 **Spring Boot Controller** 已暴露的**只读/列表类**接口，与 **Vue 前端各功能**做对照，便于实现「先查询 → 表格/卡片回显 → 再调原有写接口」的交互。  
> 依据：`src/main/java/com/library/controller/*.java`、`frontend/src/api/*.ts`、根目录 `API接口文档-前后端对接.md`。  
> **未包含**：`/test/*` 测试接口（非正式业务 API）；前端当前未接入的 `/api/notification/template/*`。

---

## 1. 图例

| 标记 | 含义 |
|------|------|
| **已有** | 后端已有可用的 GET（或 POST 但语义为「查询」如健康查询），前端可直接对接做列表/详情回显。 |
| **部分已有** | 有一部分查询能力，但关键维度仍缺接口，或现有接口存在实现缺口。 |
| **尚无** | 当前 Controller 中**无**对应只读接口，要做真实「查询回显」需后端新增（或改现有接口返回结构）。 |

| 管理端 | 故障运维 | `GET/PATCH/POST health` | 工单列表、详情 | **已有**（`GET /api/fault/list`、`GET /api/fault/{id}`；健康查询为 `POST /api/fault/health/query`） |
| 管理端 | 设备状态 | `PATCH /api/equipment/{id}/status` | 设备列表或搜索 | **尚无**（仅 PATCH，无设备分页/列表 GET） |
| 管理端 | 定时任务调试 | `POST /api/task/trigger/*` | 一般无需列表 | **不适用**（无「查询回显」强需求） |

## 3. 读者端（`/u/*`）明细

### 3.1 借书申请（`BorrowView` → `POST /api/borrow/request`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 应由登录态提供，非接口 |
| 选书（替代 `biblioId`） | **尚无** `GET` 书目检索/列表 |
| 选取书馆（替代 `preferredLibraryId`） | **尚无** 图书馆列表 |

**结论**：要做「查询回显再借书」，需新增至少 **书目查询 + 馆列表**（或合并搜索接口）。

---

### 3.2 还书（`ReturnView` → `POST /api/book/return`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 在借图书 / 条码解析副本（替代 `copyId`） | **尚无** 在借列表、条码查副本等 |
| 还书馆（替代 `returnLibraryId`） | **尚无** 馆列表 |

**结论**：要做「查在借 → 行内还书」，需新增 **当前用户借阅列表（含 copyId 等）** 与 **馆列表**（或默认馆规则由后端在列表中带出）。

---

### 3.3 取书确认（`PickupView` → `POST /api/borrow/confirm-pickup`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 当前用户 | 登录态 |
| 选一条待取书记录（替代 `borrowId`） | **尚无** 借阅/待取书列表 |

**结论**：需新增 **待取书或在借列表**（含 `borrowId`、书名、取书馆等）。

---

### 3.4 我的调拨与进度（`TransferUserView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 我的调拨列表 | `GET /api/transfer/my-transfers` | **已有**，可做表格，`records` 内字段可用于再查进度 |
| 单条进度 | `GET /api/transfer/progress/{transferId}` | **已有** |
| 批量单进度 | `GET /api/transfer/progress/batch/{orderId}` | **已有** |

**结论**：查询回显 **已具备**；前端可把「手输 transferId」改为从 `my-transfers` 的 `records` 中选（或详情抽屉再调进度 GET）。

---

### 3.5 座位（`SeatView`）

| Tab / 能力 | 方法 | 说明 |
|--------------|------|------|
| 查询可用 | `GET /api/seat/available` | **已有**，返回座位列表或自动分配结果，可直接驱动「选座」 |
| 我的预约 | `GET /api/seat/reservation/my` | **已有**，可表格 + 行内取消/签到等 |
| 取消/签到/暂离/结束 | 多个 `POST .../reservation/*` | 写接口已有；`reservationId` 可从「我的预约」行带出 |

**缺口（体验层）**：`libraryId`、可选 `areaId` 仍无统一 **馆/区字典 GET**；若不做后端，只能默认写死、或由 `available` 返回结构反推（若业务上允许单馆演示）。

**结论**：座位主流程 **查询回显条件基本满足**；**馆/区枚举** 仍缺只读 API。

---

### 3.6 故障报修（`FaultReportView` → `POST /api/fault/report`）

| 用户侧理想输入 | 后端只读能力 |
|----------------|-------------|
| 选馆/区/座/设备（替代各类 ID） | **尚无** 资源树或分页检索接口 |

说明：`GET /api/fault/list` 为**工单**列表，不是「物理馆/座/设备」主数据，不能直接替代报修对象选择。

**结论**：要做级联/搜索选目标，需新增 **馆/区/座/设备** 类只读接口（或少量聚合接口）。

## 4. 管理端（`/a/*`）明细

### 4.1 调拨与库存（`TransferAdminView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 全量调拨记录 | `GET /api/transfer/list` | **已有**，可选 `transferId` / 上下文 |
| 用户调拨（若需对照） | `GET /api/transfer/my-transfers` | **已有**（需 `userId` Query，偏运维查询） |
| 单条/批量进度 | `GET /api/transfer/progress/{transferId}`、`GET .../batch/{orderId}` | **已有** |
| 调拨建议列表 | `GET /api/transfer/suggestion/list` | **已有** |
| 调拨建议详情 | `GET /api/transfer/suggestion/{id}` | **部分已有**：实现中为 **TODO**，可能长期返回 `data=null`，需后端补齐后才适合「详情回显」 |

写操作：`POST /api/transfer/complete`、`POST /api/transfer/complete-batch`、建议审批/拒绝/触发平衡等 **已有**。

---

### 4.2 故障工单运维（`FaultManageView`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 工单分页 | `GET /api/fault/list` | **已有** |
| 工单详情 | `GET /api/fault/{id}` | **已有** |
| 资源健康 | `POST /api/fault/health/query` | **已有**（请求体带资源列表，属「查询类」） |

**结论**：管理端故障模块 **查询回显已较完整**。

---

### 4.3 设备状态（`EquipmentView` → `PATCH /api/equipment/{id}/status`）

| 能力 | 方法 | 说明 |
|------|------|------|
| 选设备再改状态 | — | **尚无** `GET` 设备列表/按馆筛选 |

**结论**：要做「列表选设备再 PATCH」，需新增 **设备分页或列表 GET**。

---

### 4.4 定时任务调试（`DevTasksView`）

仅 `POST /api/task/trigger/*`，无列表依赖。

**结论**：**不适用**「查询回显」；最多展示上次返回文案（前端状态即可）。

## 5. 后端已有 GET / 查询类接口速查（便于排期）

以下为当前 **业务 `/api/*`** 下、与「读模型/列表」相关的接口（不含仅 POST 的写路径）：

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/api/seat/available` | 可用座位 / 自动分配 |
| GET | `/api/seat/reservation/my` | 我的座位预约分页 |
| GET | `/api/transfer/progress/{transferId}` | 单条调拨进度 |
| GET | `/api/transfer/progress/batch/{orderId}` | 批量调拨进度 |
| GET | `/api/transfer/my-transfers` | 用户调拨分页 |
| GET | `/api/transfer/list` | 全量调拨分页（管理） |
| GET | `/api/transfer/suggestion/list` | 调拨建议列表 |
| GET | `/api/transfer/suggestion/{id}` | 调拨建议详情（**实现待补**） |
| GET | `/api/fault/list` | 工单分页 |
| GET | `/api/fault/{id}` | 工单详情 |
| POST | `/api/fault/health/query` | 资源健康（查询语义） |

## 6. 建议的后端补全优先级（仅建议，非实现承诺）

若目标优先打通 **读者端主路径**：

1. **在借/待取书列表**（同时支撑还书选 `copyId`、取书选 `borrowId`，可视数据模型拆成 1～2 个接口）  
2. **图书馆列表**（借书、还书、座位选馆）  
3. **书目检索**（借书选 `biblioId`）  
4. **副本解析**（可选：条码 → `copyId`，便于还书扫码）  
5. **设备列表 GET**（管理端设备维护）  
6. **馆/区/座/设备** 只读树或检索（读者报修）  
7. 补齐 **`GET /api/transfer/suggestion/{id}`** 详情实现
