# 测试指南

## 概述
本文档提供了多智能体图书馆管理系统的测试指南，包括测试环境准备、测试场景和测试步骤。

## 一、测试环境准备

### 1. 数据库重置
执行以下命令重置数据库到初始测试状态：
```bash
mysql -u root -p1234 library_db < test_data.sql
```

### 2. 启动应用
启动Spring Boot应用：
```bash
mvn spring-boot:run
```

### 3. 访问Swagger UI
打开浏览器访问：`http://localhost:8080/swagger-ui.html`

## 三、测试场景

### 场景1：本地借阅测试
**目的**：测试用户在本馆有可用书籍时的借阅流程

**步骤**：
1. 在Swagger UI中找到 `BorrowController`
2. 调用 `/api/borrow` 接口
3. 输入以下参数：
   ```json
   {
     "userId": 1,
     "biblioId": 1,
     "preferredLibraryId": 1
   }
   ```
   - userId: 1（张三）
   - biblioId: 1（Java编程思想，理科馆有3本）
   - preferredLibraryId: 1（理科馆）

**预期结果**：
- 借阅成功
- 状态为 `BORROWING`
- 无需调拨
- 用户当前借阅数+1

**验证点**：
- ✅ 借阅记录创建成功
- ✅ 副本状态更新为 `BORROWED`
- ✅ 库存数量-1
- ✅ 用户借阅计数+1

---

### 场景2：跨馆调拨测试
**目的**：测试用户在本馆无可用书籍时的跨馆调拨流程

**步骤**：
1. 在Swagger UI中找到 `BorrowController`
2. 调用 `/api/borrow` 接口
3. 输入以下参数：
   ```json
   {
     "userId": 1,
     "biblioId": 7,
     "preferredLibraryId": 1
   }
   ```
   - userId: 1（张三）
   - biblioId: 7（红楼梦，理科馆无，文科馆有4本）
   - preferredLibraryId: 1（理科馆）

**预期结果**：
- 触发跨馆调拨
- 状态为 `TRANSFERRING`
- 从文科馆调拨到理科馆
- 30分钟后状态变为 `RESERVED`

**验证点**：
- ✅ 借阅记录创建成功（状态为 `TRANSFERRING`）
- ✅ 调拨记录创建成功
- ✅ 副本状态更新为 `IN_TRANSIT`
- ✅ 文科馆库存-1
- ✅ 用户借阅计数+1

---


### 场景2.5：工程馆跨馆调拨测试（验证智能体配置）
**目的**：测试从工程馆跨馆调拨的流程，验证工程馆的智能体是否正常工作

**步骤**：
1. 在Swagger UI中找到 `BorrowController`
2. 调用 `/api/borrow` 接口
3. 输入以下参数：
   ```json
   {
     "userId": 1,
     "biblioId": 1,
     "preferredLibraryId": 2
   }
   ```
   - userId: 1（张三）
   - biblioId: 1（Java编程思想）
   - preferredLibraryId: 2（文科馆）

**说明**：
- 理科馆有3本《Java编程思想》
- 工程馆有2本《Java编程思想》
- 文科馆没有《Java编程思想》

**预期结果**：
- UserDemandAgent会向所有InventoryAgent（lib1、lib2、lib3）发送CFP消息
- lib1（理科馆）和lib3（工程馆）会返回PROPOSAL
- lib2（文科馆）不会返回PROPOSAL（因为没有这本书）
- 系统会根据评分算法选择最优方案
- 可能触发从工程馆到文科馆的调拨

**验证点**：
- ✅ lib3（工程馆）参与智能体协商
- ✅ 工程馆的库存被正确评估
- ✅ 跨馆调拨成功执行（如果选择工程馆）
- ✅ 工程馆库存-1
- ✅ 文科馆库存+1（调拨完成后）

**重要提示**：
- 此场景验证工程馆的智能体配置是否正确
- 如果工程馆未注册为智能体，系统将无法从工程馆调拨书籍
- 请查看JADE GUI确认lib3是否正常启动

### 场景3：预约排队测试
**目的**：测试多用户同时借阅稀缺书籍时的预约排队机制

**步骤**：
1. 重置数据库：`mysql -u root -p1234 library_db < test_data.sql`
2. 用户1借阅《资治通鉴》（biblioId=10，只有2本）：
   ```json
   {
     "userId": 1,
     "biblioId": 10,
     "preferredLibraryId": 1
   }
   ```
3. 用户2借阅《资治通鉴》：
   ```json
   {
     "userId": 2,
     "biblioId": 10,
     "preferredLibraryId": 2
   }
   ```
4. 用户3借阅《资治通鉴》：
   ```json
   {
     "userId": 3,
     "biblioId": 10,
     "preferredLibraryId": 1
   }
   ```

**预期结果**：
- 用户1：借阅成功
- 用户2：借阅成功
- 用户3：进入预约队列（状态为 `PENDING`）

**验证点**：
- ✅ 用户1、2借阅成功
- ✅ 用户3创建预约记录
- ✅ 预约排队顺序正确

**验证点**：

- ✅ 库存数量+1
- ✅ 用户借阅计数-1
- ✅ 预约队列处理正确

### 场景5：并发测试
**目的**：测试多用户同时借阅同一本书的并发安全性

**步骤**：
1. 重置数据库
2. 使用Postman或JMeter同时发送多个借阅请求
3. 请求参数：
   ```json
   {
     "userId": 1,
     "biblioId": 10,
     "preferredLibraryId": 2
   }
   ```
4. 同时发送3个请求（userId分别为1、2、3）

**预期结果**：
- 只有2个请求成功（因为只有2本书）
- 第3个请求进入预约队列

**验证点**：
- ✅ 并发安全性验证
- ✅ 分布式锁生效
- ✅ 数据一致性保证

### 场景7：状态转换验证
**目的**：验证状态转换的正确性

**步骤**：
1. 执行场景1（本地借阅）

**预期结果**：
- 状态转换符合规则
- 不出现非法状态转换

**验证点**：
- ✅ 借阅记录状态转换正确
- ✅ 副本状态转换正确
- ✅ 无非法状态转换

### 场景8：配置属性测试
**目的**：验证动态配置是否生效

**步骤**：
1. 修改 `application-business-rules.yml` 中的配置：
   ```yaml

     business-rules:
       borrow:
         default-borrow-days: 60  # 修改为60天
   ```
2. 重启应用
3. 执行借阅操作
4. 查询借阅记录的due_time

**预期结果**：
- 借阅期限为60天

**验证点**：
- ✅ 配置属性生效
- ✅ 业务规则动态调整

## 五、测试注意事项


### 5.2 日志查看

```bash

```

- 启动应用后会自动打开JADE GUI

```sql

SELECT * FROM tb_book_transfer ORDER BY create_time DESC;

-- 查看用户借阅统计
```

## 七、常见问题

### Q1: Agent启动失败
**A**: 检查Spring容器是否完全初始化，查看日志中的初始化顺序。

### Q2: 调拨超时
**A**: 检查定时任务是否正常执行，查看调拨记录的complete_time。

### Q3: 并发测试失败
**A**: 确认DistributedLock是否正确配置，检查锁的释放。

### Q4: 配置不生效
**A**: 确认配置文件路径正确，重启应用使配置生效。
