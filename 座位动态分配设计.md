# 座位动态分配功能设计

本文档对「座位动态分配」做综合设计，包括：功能定位、与借书/取书是否联动、表结构修订建议、动态分配策略、接口与流程，以及对 UserDemandAgent 中“借书完成后预留座位”TODO 的结论。

---

## 一、目标与原则

### 1.1 功能定位

- **座位动态分配**：在用户给定馆、日期、时段（及可选偏好）下，由系统**推荐或自动分配**可用座位，而不是仅支持“用户自选某个座位号”的静态预约。
- **与现有表结构的关系**：在现有 `tb_seat_area`、`tb_seat`、`tb_seat_reservation` 基础上扩展字段与约束，并新增业务规则与查询逻辑，不推翻现有三张表的核心含义。

### 1.2 设计原则

- **先独立后联动**：座位预约作为独立能力先闭环（查空位、预约、签到、释放），再通过“推荐入口”与借书/取书衔接，避免强耦合导致借书失败影响座位或反之。
- **可配置、可关停**：是否在借书/取书结果中展示“推荐预约座位”、是否启用自动分配，由配置或开关控制，便于分阶段上线。
- **不引入复杂中间件**：与书籍跨馆调配一致，不依赖 Redis/消息队列，用数据库 + 定时任务即可。

## 二、关于“借书的同时是否预留座位”

### 2.1 UserDemandAgent 中的 TODO

当前在**本地借阅（LOCAL_LOAN）**成功后有一处 TODO：`//TODO: 预留座位功能（Phase 2）`。  
需要明确：是“借书成功**即自动**为用户占座”，还是“借书成功**后推荐**用户去预约座位”。

### 2.2 结论：**不采用借书同时自动预留座位，改为“可选推荐”**

| 维度 | 说明 |
|------|------|
| **行为** | 借书/取书接口**不自动**创建座位预约；在返回结果中增加**可选**的“推荐预约座位”信息（如目标馆、推荐日期、预约入口），由用户**主动**选择是否预约及时段。 |
| **原因简述** | ① 用户可能不到馆或晚到，自动占座易造成座位空置、他人无法使用；② 调拨场景下用户取书时间不确定（24 小时内），自动占座时段难以设定；③ 本地借阅虽是即时取书，但“是否当天在馆自习”由用户决定，强制占座增加 NO_SHOW 与信用处理成本。 |
| **推荐策略** | 本地借阅：可推荐“当前馆 + 当日”的时段块；调拨到货/预约兑现：可推荐“取书馆 + 取书当日”的时段块；仅作推荐，不写库，用户点击后再走座位预约接口。 |

### 2.3 对 TODO 的具体建议

- **不建议**：在 `UserDemandAgent` 借书成功后直接调用“创建座位预约”的逻辑。
- **建议**：  
  - 借书/取书接口在返回体中增加**可选字段**，例如：`recommendSeatLibraryId`、`recommendSeatDate`、`recommendSeatReason`（如 `LOCAL_LOAN` / `TRANSFER_ARRIVED`），以及前端可用的“预约座位”链接或动作标识。  
  - 前端/客户端根据这些字段展示“可预约座位”入口，用户点击后再调用**座位预约/动态分配**接口，完成选时段、系统分配或自选座位、落库。  
这样既满足“借书+座位”的一体化体验，又避免自动占座带来的空置与违约问题。

## 三、表结构修订建议

在**不改变现有表核心含义**的前提下，做最小必要扩展，以支持动态分配与后续与借书联动分析。

### 3.1 保持不变的表

- **tb_seat_area**：不变。已有所属馆、开放时间、电源、状态等，足够用于筛选区域。
- **tb_seat**：不变。已有区域、编号、电源、状态，可通过 `area_id` 关联到馆与区域。

### 3.2 建议修改：tb_seat_reservation（座位预约表）

| 变更类型 | 字段名 | 类型 | 说明 |
|----------|--------|------|------|
| 新增 | library_id | bigint, NULL 允许 | 冗余所属馆，便于按馆查“某日某时段”预约、与取书馆一致；可从 seat→area 反查，但冗余可简化查询与统计。 |
| 新增 | borrow_id | bigint, NULL 允许 | 关联 tb_book_borrow.id；若本次预约来自“借书/取书推荐”则填写，便于统计与后续“到馆取书+座位”联动。 |
| 新增 | source | varchar(20), NULL 允许 | 预约来源：WALK_IN（直接预约）、BORROW_PICKUP（借书/取书推荐后预约）等，便于运营分析。 |
| 约束/索引 | - | - | 同一 seat_id + reservation_date，与 status=ACTIVE 的记录不允许 [start_time, end_time] 重叠（见下节）。 |

**DDL 示例（执行前请确认表名与现有库一致）**：

```sql
-- 新增列（若已有则跳过）
ALTER TABLE tb_seat_reservation ADD COLUMN library_id BIGINT NULL COMMENT '所属馆ID(冗余)';
ALTER TABLE tb_seat_reservation ADD COLUMN borrow_id BIGINT NULL COMMENT '关联借阅ID(借书/取书推荐预约时填写)';
ALTER TABLE tb_seat_reservation ADD COLUMN source VARCHAR(20) NULL COMMENT '预约来源:WALK_IN/BORROW_PICKUP';

-- 便于按馆、按借阅查预约
CREATE INDEX idx_seat_reservation_library_date ON tb_seat_reservation(library_id, reservation_date);
CREATE INDEX idx_seat_reservation_borrow ON tb_seat_reservation(borrow_id);
```

**说明**：

- `library_id`：创建预约时由 `seat → area → library_id` 写入。
- `borrow_id`：仅当用户从“借书/取书推荐”入口进入并确认预约时写入。
- 重叠校验在应用层或通过存储过程/触发器实现；若仅应用层校验，需在预约写入前查询该座位当日 ACTIVE 预约并判断时间段不重叠。

### 3.3 唯一与重叠约束（业务规则）

- **唯一性**：同一 `(seat_id, reservation_date)` 下，任意两条 `status = 'ACTIVE'` 的记录的 `[start_time, end_time]` 不得重叠。
- 实现方式二选一或并存：  
  - **应用层**：在“创建座位预约”服务中，先按 seat_id + reservation_date 查 ACTIVE，再判断新区间是否与已有区间重叠，再 insert。  
  - **数据库**：若使用 MySQL 8.0+，可考虑用表达式唯一索引或 CHECK 约束（若支持）辅助；否则仍以应用层为准。

### 3.4 可选：业务规则配置

在 `application-business-rules.yml`（或等价配置）中增加座位相关配置，便于后续调整，例如：

```yaml
library:
  business-rules:
    seat:
      # 单次预约最小时长（分钟）
      min-duration-minutes: 60
      # 单次预约最大时长（分钟），如 4 小时
      max-duration-minutes: 240
      # 预约时段步长（分钟），如 120 表示只能约 10:00-12:00、12:00-14:00
      slot-step-minutes: 120
      # 最早可预约日期（相对今天的天数），如 0 表示仅当天
      earliest-booking-days: 0
      # 最晚可预约日期（相对今天的天数），如 7 表示最多约到 7 天后
      latest-booking-days: 7
      # 未签到多少分钟后视为 NO_SHOW 并释放座位
      no-show-after-minutes: 30
```

初期也可在代码中用常量实现，待稳定后再迁入配置。

## 四、动态分配策略（核心逻辑）

### 4.1 输入

- **必填**：馆 ID（library_id）、预约日期（reservation_date）、开始时间（start_time）、结束时间（end_time）。
- **可选**：区域 ID（area_id）、是否需要电源（has_power）、是否“仅系统分配一个座位”（auto_assign=true）。

### 4.2 流程概要

1. **按馆 + 日期 + 时段筛出可用座位**  
   - 取该馆下所有 `tb_seat`（通过 `tb_seat_area.library_id` 关联），且 `status = 'AVAILABLE'`（或结合实时占用见下）。  
   - 排除在该 `reservation_date` 下已有 ACTIVE 预约且时间与 [start_time, end_time] 重叠的 seat_id。  
   - 若传入 area_id，则只保留该区域的座位；若传入 has_power=true，则只保留 has_power=1 的座位。
2. **排序与推荐**  
   - 可按“区域楼层、座位号”等稳定排序，便于前端展示一致；若有历史占用率统计，也可按“优先推荐冷门座”等策略排序（后期增强）。  
   - 若 `auto_assign=true`，取排序后的第一个座位作为“系统分配”结果；否则返回列表供用户选择。
3. **写入预约**  
   - 插入 `tb_seat_reservation`，设置 library_id、seat_id、user_id、reservation_date、start_time、end_time、status=ACTIVE；若有 borrow_id/source，一并写入。  
   - 将对应 `tb_seat.status` 置为 OCCUPIED（或保留 AVAILABLE，仅用预约表约束占用，视产品选择；若置为 OCCUPIED，需在签到/取消/超时释放时改回 AVAILABLE）。

### 4.3 座位状态与预约的关系（建议）

- **方案 A**：座位仅有 AVAILABLE/OCCUPIED/DISABLED；一旦生成 ACTIVE 预约即将座位置为 OCCUPIED，取消或 NO_SHOW 后置回 AVAILABLE。  
- **方案 B**：座位始终为 AVAILABLE，仅用预约表 + 时间重叠约束表示“已被预约”；查询“当前是否可用”时，根据当前时间与 ACTIVE 预约的 start_time/end_time 判断。  
推荐**方案 A**，与“座位表 status 即当前是否可用”的语义一致，便于馆内大屏或简单查询“当前空座数”。

## 五、接口与流程设计

### 5.1 座位动态分配/查询可用座位

- **接口**：`GET /api/seat/available` 或 `POST /api/seat/allocate`（若希望用 POST 传 body）。  
- **参数**：libraryId、reservationDate、startTime、endTime；可选 areaId、hasPower、autoAssign（为 true 时返回单个推荐座位，否则返回列表）。  
- **返回**：座位列表（含 seatId、areaId、seatNo、hasPower、areaName 等）或单个分配结果；若无可选座位返回空列表或 4xx。

### 5.2 创建座位预约

- **接口**：`POST /api/seat/reserve`。  
- **参数**：userId、seatId、reservationDate、startTime、endTime；可选 borrowId、source。  
- **校验**：该座位在该日期、该时段无 ACTIVE 重叠；start/end 在馆/区域开放时间内；时长在配置的 min/max 之间。  
- **落库**：插入 tb_seat_reservation（library_id 由 seat→area 带出），并将 tb_seat.status 置为 OCCUPIED（若采用方案 A）。

### 5.3 取消预约

- **接口**：`POST /api/seat/reservation/cancel`。  
- **参数**：reservationId、userId（校验本人）。  
- **逻辑**：将预约状态置为 CANCELED；若采用方案 A，将对应座位置回 AVAILABLE。

### 5.4 签到

- **接口**：`POST /api/seat/reservation/check-in`。  
- **参数**：reservationId、userId。  
- **逻辑**：校验预约存在、ACTIVE、未过期（当前时间在 [start_time, end_time] 内或允许一定迟到时间），写入 check_in_time，状态可改为 COMPLETED 或保持 ACTIVE 至时段结束（由产品定）。

### 5.5 超时未签到释放（NO_SHOW）

- **定时任务**：按配置的 `no-show-after-minutes`，扫描 ACTIVE 且未填写 check_in_time、且 start_time 已过超过 N 分钟的预约，将状态改为 NO_SHOW，并将座位置回 AVAILABLE。  
- **可选**：与图书预约类似，对 NO_SHOW 做信用扣分或记录，便于后续限制频繁违约用户。

### 5.6 我的座位预约列表

- **接口**：`GET /api/seat/reservation/my?userId=xxx&status=ACTIVE&fromDate=xxx&toDate=xxx`。  
- **返回**：当前用户的座位预约列表，含馆、区域、座位号、日期时段、状态等，便于用户取消或签到。

### 5.7 与借书/取书结果的联动（推荐信息）

- **借书**（BorrowController / UserDemandAgent 返回体）：在策略为 LOCAL_LOAN 时，增加 `recommendSeatLibraryId`、`recommendSeatDate`（如当天）、`recommendSeatReason": "LOCAL_LOAN"`，以及前端用的 `actionRecommendSeat: true`。  
- **取书**（调拨完成/取书确认后）：在返回体中增加 `recommendSeatLibraryId`（取书馆）、`recommendSeatDate`（取书日或次日）、`recommendSeatReason": "TRANSFER_PICKUP"` 等。  
- 前端根据上述字段展示“预约座位”按钮或链接，跳转到座位预约页并预填馆、日期，用户选择时段并调用 `GET /api/seat/available` 与 `POST /api/seat/reserve`，可选传 borrowId 与 source。

## 六、与智能体的关系（是否由 Agent 执行）

- **建议**：座位预约与动态分配**不放入 UserDemandAgent 的流程内部**执行。  
- **理由**：  
  - 座位是“用户到馆行为”，与“是否借到书”解耦更清晰；  
  - 占座逻辑、时间重叠校验、定时释放等更适合放在 Service 层统一处理，便于测试与运维；  
  - Agent 仅负责在借书/取书结果中**附带推荐信息**（如 recommendSeatLibraryId、recommendSeatDate），由前端/客户端再调座位相关接口。  
- 若后续希望“由某个 Agent 根据馆内实时人数推荐座位”，可单独增加 SeatRecommendationAgent，输入馆+日期+时段，输出推荐座位列表，仍由现有座位预约服务负责落库与状态维护。

## 七、实施顺序建议

1. **表结构**：按第三节修改 `tb_seat_reservation`（library_id、borrow_id、source），并实现应用层“同一座位同日期 ACTIVE 时段不重叠”的校验。  
2. **核心服务与接口**：实现可用座位查询、创建预约、取消、签到、我的预约列表；定时任务 NO_SHOW 释放（及可选信用扣分）。  
3. **动态分配**：在“查询可用座位”上封装“推荐排序 + 可选 autoAssign”，对外作为“动态分配”能力。  
4. **借书/取书联动**：在借书与取书返回体中增加推荐座位字段与前端入口，不自动写座位预约。  
5. **UserDemandAgent**：移除“借书成功后自动预留座位”的 TODO，改为在返回结果中设置推荐座位相关字段（或由 Controller 层根据策略统一追加）。

## 八、小结

| 项目 | 结论 |
|------|------|
| **借书同时是否预留座位** | **不自动预留**；改为在借书/取书结果中返回**推荐预约座位的入口与参数**，由用户主动预约。 |
| **表结构** | 在 `tb_seat_reservation` 增加 library_id、borrow_id、source；其余表可保持不变。 |
| **动态分配** | 按馆+日期+时段+可选区域/电源筛出可用座位，排序后返回列表或单座；创建预约时校验时间不重叠并在应用层保证。 |
| **与 Agent** | 座位预约与释放不放在 UserDemandAgent 内执行；Agent/接口层仅提供“推荐座位”信息，实际预约走独立接口与服务。 |

按上述设计可实现完整的“座位动态分配”闭环，并与借书/取书形成**可选联动**，同时避免自动占座带来的空置与违约问题。

## 九、典型用户流程（简要）

| 场景 | 流程 |
|------|------|
| **纯预约座位** | 用户选择馆、日期、时段（及偏好）→ 调用“查询可用座位”→ 选择座位或由系统分配 → 调用“创建预约”→ 到馆后“签到”；未签到则由定时任务 NO_SHOW 释放。 |
| **借书后想占座** | 用户借书成功（本地/调拨）→ 接口返回带“推荐预约座位”信息 → 用户点击“预约座位”→ 跳转预约页（预填馆、推荐日期）→ 选时段并创建预约 → 同上。 |
| **取消/违约** | 用户主动取消 → 调用取消接口 → 座位置回 AVAILABLE；超时未签到 → 定时任务将预约置 NO_SHOW、座位置回 AVAILABLE（可选扣信用分）。 |

实施时请同步更新《数据库结构.md》中 tb_seat_reservation 的字段说明。

## 十、开发前确认清单

在动手开发前，建议与产品/前端对齐以下项，避免实现与预期不一致。

### 1. 座位状态与时段结束

- **预约生效时**：创建 ACTIVE 预约后，是否立刻将 `tb_seat.status` 置为 `OCCUPIED`？（设计推荐：是。）
- **时段结束后**：到达 `end_time` 时，是否由定时任务自动将预约置为 `COMPLETED` 并把座位置回 `AVAILABLE`？还是仅依赖“签到后即 COMPLETED、未签到则 NO_SHOW”？（若采用“时段结束也释放”，需要增加定时任务扫描 end_time 已过的 ACTIVE 预约。）

### 2. 签到后的状态

- 用户到馆点击“签到”后，预约状态是**立即改为 COMPLETED**，还是**保持 ACTIVE 直到 end_time 再由定时任务改为 COMPLETED**？  
  （若立即 COMPLETED，座位是否在签到时就置回 AVAILABLE？一般建议：签到后仍占座到 end_time，仅状态可标为“已签到”，到 end_time 再释放座位，便于统计“实际使用时长”。）

### 3. 开放时间校验

- 校验“预约时段在开放时间内”时，以**馆的 open_time** 为准、**区域的 open_time** 为准，还是**两者都要满足**？
- `open_time` 的格式是否统一为 `"HH:mm-HH:mm"`（如 `"08:00-22:00"`），是否需要支持多段（如午休闭馆）？

### 4. 同一用户同一时段多座

- 是否允许**同一用户**在同一馆、同一天、**同一时间段**预约**多个座位**（如帮他人占座）？  
  - 若**不允许**：创建预约时需校验“该用户在该馆、该日期、与该时段重叠的 ACTIVE 预约仅允许一条”。

### 5. NO_SHOW 与信用分

- 座位预约 **NO_SHOW**（超时未签到）是否要**扣用户信用分**？若扣，扣多少？是否与图书预约逾期共用 `tb_user.credit_score`？  
- 若需要扣分，是否在业务配置中增加一项（如 `seat.no-show-credit-deduction: 2`）？

### 6. 借书/取书联动时的 borrowId 与 source

- 用户从“借书/取书推荐”入口进入并预约时，**前端是否会传 borrowId**？若传，后端是否要校验该 borrow 属于当前用户且状态/取书馆与预约馆一致？
- **source** 由**前端传入**（如 `BORROW_PICKUP`）还是**后端根据是否带 borrowId 自动设置**？

### 7. 查询可用座位的默认行为

## 十一、暂离与结束使用设计

### 11.1 是否需要“暂离 / 结束使用”接口（评估结论）

- **建议增加**：  
  - **暂离接口**：允许用户在已经签到、正在使用座位的过程中，短时间离开（如吃饭、上厕所），期间不被系统释放座位，也不允许他人抢占。  
  - **结束使用接口**：允许用户在预约结束时间之前主动释放座位，提升资源利用率。
- 你的需求“中午、晚上可暂离 2 小时，其余时间 30 分钟”非常合理，适合通过**业务配置 + 定时任务**实现，无需复杂算法。

### 11.2 数据与状态设计

- **不新增复杂状态机字段**，继续保持 `SeatReservation.status` 只表示整体生命周期：`ACTIVE/COMPLETED/CANCELED/NO_SHOW`。  
- 引入一个**使用中间状态**的抽象，在实现上可用内存状态 + 定时任务，也可扩展字段（如后续需要持久化时）：  
  - `IN_USE`：已签到且当前在座（默认状态）；  
  - `TEMP_LEAVE`：已签到但暂离中，尚在允许时长内；  
  - `FINISHED`：用户主动结束使用或预约自然结束（对应 `status=COMPLETED`）。
- 初期实现建议：  
  - `IN_USE` / `TEMP_LEAVE` 只在服务层内存中维护（比如通过缓存或本地 Map），配合定时任务检查是否超时；  
  - 真正对外可见/可统计的仍以预约记录 (`status` + `check_in_time` + `start/end_time`) 为准，避免频繁改字段。

若你更偏向“所有状态都落库”，可在后续迭代中考虑在 `tb_seat_reservation` 增加：  

- `temp_leave_start_time`：最近一次暂离开始时间；  
- `temp_leave_expire_time`：本次暂离最晚应返回时间（由规则计算）；  
- 或增加 `usage_status` 字段保存 `IN_USE/TEMP_LEAVE` 等，但当前不是必须。

### 11.3 业务规则：暂离时长

建议将你的规则固化为配置（可在 `application-business-rules.yml` 中扩展）：  

```yaml
library:
  business-rules:
    seat:
      # 默认暂离允许时长（分钟）
      temp-leave-default-minutes: 30
      # 中午饭点暂离规则
      temp-leave-lunch:
        start: "11:30"
        end: "13:30"
        max-minutes: 120
      # 晚饭时间暂离规则
      temp-leave-dinner:
        start: "17:30"
        end: "19:30"
        max-minutes: 120
```

计算逻辑示例：  

- 用户点击“暂离”时：  
  - 若当前时间处于 lunch/dinner 区间，则以对应的 `max-minutes` 作为本次暂离上限；  
  - 否则使用 `temp-leave-default-minutes`；  
  - 同时受限于**预约结束时间**：本次暂离最晚返回时间 = `min(当前时间 + 上述分钟, reservation end_time)`。

### 11.4 接口设计

#### 11.4.1 暂离

- **接口**：`POST /api/seat/reservation/temp-leave`  
- **参数**：`reservationId`、`userId`。  
- **前置条件**：  
  - 预约存在、`status=ACTIVE`；  
  - 用户已签到（`check_in_time` 非空）；  
  - 当前未处于已记录的暂离中（避免重复点击）。  
- **核心行为**：  
  - 按上一小节规则计算本次暂离的“最迟返回时间”；  
  - 在服务层记录该预约的暂离开始时间和最迟返回时间；  
  - 不改变 `tb_seat.status`，仍保持 `OCCUPIED`，防止他人被分配到该座。

#### 11.4.2 结束使用

- **接口**：`POST /api/seat/reservation/finish`  
- **参数**：`reservationId`、`userId`。  
- **核心行为**：  
  - 校验预约为当前用户、`status=ACTIVE`；  
  - 将预约状态置为 `COMPLETED`；  
  - 将关联座位置回 `AVAILABLE`；  
  - 清除与该预约相关的暂离状态记录（若有）。

### 11.5 定时任务：暂离超时与自然结束

在原本“NO_SHOW 释放”任务之外，增加/扩展一个定时代码逻辑：  

1. **处理暂离超时**：  
   - 扫描当前处于 `TEMP_LEAVE` 状态的预约（或从内存记录中读取）；  
   - 若当前时间 > 暂离“最迟返回时间”：  
     - 将预约状态置为 `COMPLETED`（代表“使用结束且未返回”）；  
     - 将座位置回 `AVAILABLE`；  
     - 视情况记录一条“暂离超时自动释放”的日志（默认不扣信用分，仅作为运营指标）。  
2. **处理自然结束**：  
   - 对仍为 `ACTIVE` 且 `end_time` 已过的预约（即使未点击“结束使用”也算自然结束）：  
     - 若用户曾签到：可直接置为 `COMPLETED` 并释放座位；  
     - 若从未签到：已由 NO_SHOW 逻辑处理（与图书预约规则保持一致）。

### 11.6 用户体验与规则说明

- 用户视角下，“签到 → 暂离 → 回来继续用 → 结束使用 / 到点自动结束”形成完整闭环：  
  - 签到后可点击“暂离”，系统会提示“本次暂离最晚需在 XX:YY 前返回”；  
  - 在允许时长内返回，不触发任何处罚；  
  - 若超过时长仍未返回，系统自动结束使用并释放座位，后来的用户可以再次预约该座。  
- 午晚餐 2 小时暂离与其他时段 30 分钟规则，建议在前端“预约规则说明”中清晰展示，避免用户误解。

整体上，这一扩展与原有设计兼容：  

- 不改变预约核心状态枚举，仅在服务层补充“IN_USE/TEMP_LEAVE”语义；  
- 依然通过定时任务来完成“自动释放座位”，与你的“当用户结束使用或者预约时间到时，自动释放座位”的要求完全一致。
